## 1. Test Case Overview

### 1.1 Scope

This document contains test cases for VacaFlow_14 — an internal absence and vacation request management system. Test cases cover the following areas as defined in Test Plan TP-001:

- **Authentication and session management:** Registration, login, logout, current-user identity derivation
- **Employee request lifecycle:** Draft creation, editing, submission, cancellation, date validation, read-only enforcement, reason field validation
- **Manager review workflow:** Review list filtering, approval, rejection, optional comment, Approval record integrity, self-approval prohibition
- **Business rule enforcement at the API layer:** All 21 business rules from BR-001 v1.2 (BR-USER-001 through BR-USER-003, BR-REQ-001 through BR-REQ-006, BR-FIELD-001, BR-MGR-001 through BR-MGR-004, BR-APPR-001 through BR-APPR-002, BR-STATE-001, BR-SEC-001 through BR-SEC-002, BR-DATA-001 through BR-DATA-002)
- **Request lifecycle state transitions:** All five valid transitions and all six prohibited transitions
- **Authorization and identity integrity:** Session-derived identity enforcement; unauthorized operation rejection; role bypass attempts
- **Reason field boundary validation:** Missing, empty, 500-character, and 501-character scenarios (TC-FIELD-*)
- **Non-functional requirements:** Password hashing, database file protection, usability baseline, local performance, data persistence
- **End-to-end acceptance workflow:** Full Playwright E2E suite aligned to GOV-001 §6.2 acceptance script

**Test Environment:** Local execution environment with seeded SQLite database (seeded absence types: Vacation, Personal Leave, Sick Leave; seeded Manager account); application run from source code via local API (`dotnet run`) and web app (`npm run dev`).

### 1.2 Test Case Naming Convention

`TC-[Module]-[Type]-[Number]`

| Segment | Values | Description |
|---------|--------|-------------|
| Module | AUTH, EMP, MGR, APPR, FIELD, SEC, REL, USE, E2E | Feature area code |
| Type | P (Positive), N (Negative), E (Edge/Security) | Test polarity |
| Number | 001, 002, … | Sequential within module/type |

**Examples:**
- `TC-AUTH-P-001` — Authentication positive test #1
- `TC-EMP-N-002` — Employee lifecycle negative test #2
- `TC-FIELD-N-003` — Reason field boundary negative test #3
- `TC-SEC-E-001` — Security/authorization edge test #1
- `TC-E2E-P-001` — End-to-end positive workflow #1
