## 1. Executive Summary

### 1.1 Purpose

This document describes the technical architecture for VacaFlow_14 (IGS Solutions), providing a consolidated reference of requirements, constraints, quality attributes, and architecture decisions to guide the development team. It integrates content from the Strategic Intake (SI-001), Functional Requirements Specification (FRS-001), Non-Functional Requirements Specification (NFR-001), Business Rules Catalog (BR-001), Requirements Traceability Matrix (RTM-001), and Utility Tree to serve as the single authoritative architecture communication document for all stakeholders prior to detailed design.

### 1.2 Scope

This architecture covers the VacaFlow MVP application in its entirety, including: the Next.js/React web frontend, the ASP.NET Core Minimal API backend, SQLite persistence via Entity Framework Core, and application-managed authentication (registration, login, logout, current user). The architecture encompasses four business domains — Employee, Absence Type, Request, and Approval — and the full request lifecycle (Draft, Submitted, Approved, Rejected, Cancelled). The layered structure spans Domain, Application, Infrastructure, and Api layers following a reduced Onion Architecture.

**Explicitly excluded from this architecture:** Microsoft Entra ID and any corporate SSO provider, Azure deployment, Docker and CI/CD pipelines, email or Teams notifications, password reset and email verification flows, account administration screens, vacation balance and holiday calendar calculations, overlapping request validation, attachments, reporting or dashboards, HR views, multi-level approvals, approval delegation, payroll/HR/calendar/directory integrations, advanced audit logs, and automated backup mechanisms.

### 1.4 Architecture Summary

VacaFlow_14 adopts a **Reduced Onion Architecture** pattern (Domain, Application, Infrastructure, Api, Web), selected via Quality Attribute Weighting analysis. Key drivers: **Security** (weight 10, High priority — password hashing and server-side identity derivation are hard acceptance-rejection conditions), **Reliability** (weight 9, High priority — state transition integrity, date validation, and data persistence), and **Availability** (weight 7, High priority — cold-start automation and crash-free acceptance walkthrough). Trade-offs accepted: No Azure or cloud hosting for the MVP (compatibility over scalability); SQLite chosen over a server database engine (simplicity over concurrent scale); no MediatR, CQRS, or generic repository abstractions (directness over extensibility framework).

## 3. Constraints and Assumptions

### 3.1 Constraints

> Source: `sources/01-intake-strategic-intake-sources.md` — Questions 11a, 11b, 11c

#### 3.1.1 Business Constraints

- The application must run locally from source code; no Azure deployment, no cloud hosting, and no shared network server environment are permitted for the MVP.
- Blocking defects identified during the review window must be resolved before final acceptance sign-off. Cosmetic defects may be deferred.
- The absence type catalog (Vacation, Personal Leave, Sick Leave) is fixed for the MVP; no maintenance screen is provided.
- Manager assignment is established via seed data or controlled setup and does not change during the MVP lifecycle.
- The development team has local environments capable of running Next.js and ASP.NET Core simultaneously.

#### 3.1.2 Legal Constraints

- No formal regulatory compliance framework (GDPR, HIPAA, SOC 2) applies to the MVP. The application stores basic employee identity data (name, email, hashed password) and absence request data (dates, reasons, approval comments). This scope is acknowledged by the sponsor.
- If VacaFlow is promoted to a production system, privacy policy, data retention rules, and formal compliance requirements must be assessed before promotion.
- No privacy notice or consent flow is required within the MVP application.

#### 3.1.3 Technical Constraints

- The technology stack is constrained to: Next.js/React (frontend), ASP.NET Core Minimal API (backend), Entity Framework Core (ORM), and SQLite (embedded database). No alternative frameworks may be introduced.
- Passwords must be stored as cryptographic hashes (BCrypt, PBKDF2, or Argon2). Plain-text password storage is a hard rejection condition for acceptance.
- The API must always derive the current user and the responsible approver from the authenticated server-side session; the frontend must never supply a trusted employee or approver identifier for business decisions.
- Microsoft Entra ID and any corporate SSO provider are explicitly deferred for the MVP; local authentication is the required mechanism.
- Docker, CI/CD pipelines (GitHub Actions, Azure DevOps), and all cloud infrastructure are explicitly excluded.
- The codebase must follow reduced Onion Architecture layer separation (Domain, Application, Infrastructure, Api, Web) without introducing MediatR, generic repository abstractions, CQRS frameworks, or event bus patterns.
- Automatic database migration must run on startup without manual intervention; no manual schema setup step is permitted.
- The SQLite database file must not be committed to source control and must not contain real credentials.

### 3.2 Assumptions

- The application will be run locally from source code only; no network deployment, cloud hosting, or shared server environment is assumed for the MVP.
- A single SQLite database file is sufficient for the MVP's data volume and concurrent usage, given that only a small number of reviewers and test accounts will use the system during the validation window. This assumption originates from SI-001 §5 and is marked **Not Validated**; it is also an open Critical Information Gap in SI-001 §6 (number of employees expected during the review period).
- At least one Manager account will be available either through seed data or the registration flow before approval scenarios are tested.
- The absence type catalog (Vacation, Personal Leave, Sick Leave) is fixed for the MVP and does not require a maintenance screen.
- Manager assignment is established at registration time or through seed data and does not change during the MVP lifecycle; no reassignment or delegation flow is needed.
- Password hashing is the only credential security requirement for the MVP; multi-factor authentication, password reset, and email verification are not required.
- The implementation team may choose the simplest consistent session or token mechanism (cookie-based or JWT) for maintaining authenticated state, provided the server always derives user identity from it and the frontend never supplies a trusted identifier.
- No automated backups are required; the SQLite file location will be documented so it can be copied manually if needed.
- The MVP will not enforce overlapping request validation; two requests from the same employee for overlapping date ranges are permitted in this version.

## 5. Architecture Overview

### 5.1 Architecture Principles

| Principle | Description | Rationale |
|-----------|-------------|-----------|
| Dependency Rule Enforcement | All source dependencies point inward — Domain has no dependencies; Application depends only on Domain; Infrastructure depends on Domain and Application; Api depends on Application and Infrastructure solely for wiring. | Prevents inner business layers from being contaminated by infrastructure or presentation concerns, enabling the domain and use cases to remain stable when technology choices change. |
| Server-Side Identity Authority | The API always derives the current user and the responsible approver from the authenticated session or token. The frontend never sends a trusted employee or approver identifier for business decisions. | Eliminates the risk of identity spoofing from the frontend, which was identified as a key project risk. Approval responsibility and request ownership are always determined by the authenticated context, not by client-supplied values. |
| Minimal Scope Discipline | The architecture includes only the components required to validate the complete request lifecycle. No optional capabilities, speculative features, or advanced patterns are added beyond what the MVP demands. | Scope creep toward a full HR platform was explicitly identified as a primary risk. Keeping the architecture small and focused ensures the MVP can be delivered and validated without unnecessary complexity. |
| Persistence Ignorance | Domain entities carry no ORM attributes, no EF Core base class dependencies, and no knowledge of how they are stored. All object-relational mapping is configured through Fluent API in the Infrastructure layer. | Keeps the domain model clean and decoupled from the chosen persistence technology, so that storage details can change without affecting business rules or use case logic. |
| Business Rule Enforcement at the API Layer | All business rules — date validations, state transition guards, ownership checks, and manager authorization — are enforced by the API, not only by the UI. The UI may reflect these rules, but cannot be the sole enforcement point. | Ensures that rules cannot be bypassed by manipulating the frontend. Any bypass of authenticated-user identity or approval by a non-manager is a stated rejection condition for the MVP. |

### 5.2 Architecture Style

> *Auto-populated from Utility Tree sources — do not modify manually.*

**Style**: Reduced Onion Architecture (Domain, Application, Infrastructure, Api, Web)

**Rationale**: Selected via Quality Attribute Weighting analysis. The weighted quality attribute scores — Security (10), Reliability (9), Availability (7), and Maintainability (7) — drive the need for strict layer separation that isolates business rules from infrastructure and prevents identity-related risks from propagating across layers. The sponsor explicitly requested Clean Architecture principles without unnecessary patterns, making the reduced Onion structure the appropriate fit. See the Utility Tree document for the full scenario-by-scenario evaluation.

**Key drivers**:
- **Security** (priority 10) — hard rejection conditions on password hashing and server-side identity derivation require infrastructure concerns (hashing, session management) to be cleanly separated from domain logic and API routing
- **Reliability** (priority 9) — state machine integrity and date validation must be enforced in Domain/Application layers, independent of Infrastructure or UI
- **Maintainability** (priority 7) — layer separation is a testability prerequisite; the automated coverage targets (≥ 80% Domain, ≥ 70% Application) are only achievable if those layers have no Infrastructure dependencies

**Trade-offs accepted**:
- SQLite over a server database engine — simplicity and local portability are favored over concurrent scale, which is not a requirement for this MVP
- No Azure or cloud hosting — local execution is the required context; cloud scalability patterns are deferred
- No MediatR, CQRS, or generic repository abstractions — directness and reduced complexity are favored over extensibility frameworks; the sponsor explicitly prohibited these patterns

### 5.3 High-Level Architecture

VacaFlow MVP follows a reduced Onion Architecture with four backend layers. Dependencies always point inward. The Next.js frontend is a separate application communicating exclusively over HTTP.

```
┌─────────────────────────────────────────────────────┐
│                   Web (Next.js/React)                │
│  Register · Login · Employee Request List ·          │
│  Request Form · Manager Review List                  │
└───────────────────────┬─────────────────────────────┘
                        │ HTTP (REST)
┌───────────────────────▼─────────────────────────────┐
│              Api (ASP.NET Core Minimal API)           │
│  Auth Endpoints · Workflow Endpoints                  │
│  Middleware (Exception handling, Identity resolution) │
│  ┌───────────────────────────────────────────────┐   │
│  │              Application                       │   │
│  │  Use Case Services · Interfaces/Ports ·        │   │
│  │  Business Rules · DTOs                         │   │
│  │  ┌─────────────────────────────────────────┐  │   │
│  │  │            Infrastructure                │  │   │
│  │  │  EF Core · SQLite · Repository/UoW ·     │  │   │
│  │  │  Auth (hashing, session/token) ·         │  │   │
│  │  │  Seed Data                               │  │   │
│  │  │  ┌───────────────────────────────────┐   │  │   │
│  │  │  │           Domain                  │   │  │   │
│  │  │  │  Entities: Employee, AbsenceType, │   │  │   │
│  │  │  │  Request, Approval                │   │  │   │
│  │  │  │  Enums: RequestStatus, Role       │   │  │   │
│  │  │  │  Exceptions · Business Constants  │   │  │   │
│  │  │  └───────────────────────────────────┘   │  │   │
│  │  └─────────────────────────────────────────┘  │   │
│  └───────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
                        │
              ┌─────────▼──────────┐
              │   SQLite Database  │
              │  (local file)      │
              └────────────────────┘
```

**Key interaction flows:**

- The Next.js frontend communicates with the API exclusively over HTTP. It renders state received from the API and never makes business decisions about identity or authorization.
- The Api layer receives HTTP requests, resolves the authenticated user from the session/token, enforces business rules, and delegates to Application services.
- Application services orchestrate use cases using domain entities and infrastructure ports (repository interfaces). They enforce state transition rules, ownership checks, and manager authorization.
- Infrastructure implements persistence (EF Core + SQLite), the repository and Unit of Work, authentication support (password hashing, session or token issuance), and seeds the initial absence types and manager account.
- Domain contains the four core entities (Employee, AbsenceType, Request, Approval), the RequestStatus and Role enumerations, domain exceptions, and business constants. It has no dependencies on any other layer.

### 5.4 Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Authentication mechanism | Application-managed local registration and login with hashed passwords | Microsoft Entra ID and corporate SSO are explicitly deferred for the MVP. Local authentication eliminates the previous risk of frontend-supplied user identities and satisfies the requirement that the API always derives the current user from the authenticated session. |
| Database technology | SQLite via Entity Framework Core | The MVP runs locally from source code with no Azure deployment, Docker, or cloud hosting. SQLite keeps the application fully self-contained, requires no server setup, and is sufficient for the small user count and local execution context of the MVP. |
| Backend framework and API style | ASP.NET Core Minimal API | Confirmed by the sponsor and functional analyst as the appropriate backend for this MVP. Minimal API avoids unnecessary scaffolding and aligns with the principle of not adding architectural complexity beyond what the MVP needs. |
| Frontend technology | Next.js and React | Confirmed in scope from the earliest meetings. Provides the web interface for employee and manager workflows with a compact, functional UI limited to the screens required to complete the request lifecycle. |
| Architecture pattern | Reduced Onion Architecture (Domain, Application, Infrastructure, Api) | The sponsor explicitly requested Clean Architecture principles without unnecessary patterns such as MediatR, CQRS, generic repositories beyond the basics, or large frameworks. The reduced Onion structure separates domain concepts, use cases, infrastructure, and API endpoints while keeping the implementation small. |
| Identity resolution for business operations | API derives employee and approver identity exclusively from the authenticated session | The frontend must not send a trusted employee or approver identifier. Allowing the frontend to supply these values was the primary authentication risk identified in the project. All create, submit, cancel, approve, and reject operations use the server-side authenticated user. |
| Manager assignment model | Simple manager reference stored on the Employee record, established via seed or controlled setup | The MVP does not require a full organizational chart. The API checks this assignment when validating approval authorization, without an admin UI or directory integration. Formal confirmation of the one-to-one model is recorded as an open item in SI-001 §6. |
| Seed data strategy | Seed absence types (Vacation, Personal Leave, Sick Leave) and at least one manager account at database initialization | No catalog maintenance screen is included in the MVP. Seeding ensures a reviewer can run the API and immediately execute the full workflow without manual data setup. |
