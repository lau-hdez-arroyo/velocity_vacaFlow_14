## 1. Overview

### 1.1 Purpose

This document specifies the infrastructure requirements for VacaFlow_14, an internal absence and vacation request management system developed for IGS Solutions. It defines compute, persistence, networking, and security requirements necessary to support local development and acceptance validation of the MVP.

### 1.2 Scope

VacaFlow_14 is a **local-execution MVP**. The application runs entirely from source code on a reviewer's machine. There are no cloud environments, no external infrastructure services, and no containerized deployments in scope for this phase.

This specification covers:

- Local machine compute and runtime prerequisites
- SQLite file-based persistence and storage requirements
- Localhost network configuration and inter-process communication
- Security controls applicable to a local execution context
- Monitoring and logging mechanisms available without external infrastructure
- Startup, seeding, and recovery procedures

Cloud infrastructure (AWS, Azure, GCP), Docker, CI/CD pipelines, and production-grade deployment are **explicitly out of scope** for this MVP (NFR-COMPAT-001).

### 1.3 Execution Model

| Dimension | Value |
|-----------|-------|
| Hosting model | Local execution from source code only |
| Cloud provider | None тАФ On-premises / developer workstation |
| Environments provisioned | None (no Dev/QA/Staging/Prod cloud environments) |
| Containerization | Not used; Docker explicitly excluded |
| External database server | None; SQLite embedded via EF Core |
| External services | None |
| Access model | Localhost loopback only |

> This profile is derived from SAD-DD-001 ┬з1.1, ┬з2.2 (Container Diagram), and NFR-001 v3.0 NFR-COMPAT-001. The two-command startup model (`dotnet run` + `npm run dev`) is the complete provisioning procedure for VacaFlow_14.

## 3. Compute Requirements

### 3.1 Backend тАФ ASP.NET Core API

The ASP.NET Core API process runs as a foreground `dotnet run` process on the local machine. No process manager, container orchestrator, or daemon configuration is required.

| Attribute | Value |
|-----------|-------|
| Framework | .NET 8 (ASP.NET Core Minimal API) |
| Process model | Single foreground process (`dotnet run`) |
| Default port | `http://localhost:5000` (or as configured in `launchSettings.json`) |
| Auto-scaling | Not applicable (single local process) |
| Load balancing | Not applicable |
| CPU allocation | Shared with the OS; no dedicated allocation required |
| Memory footprint (estimated) | 80тАУ200 MB RAM at idle; up to 400 MB under active walkthrough load |
| Startup time | < 10 seconds cold start including EF Core migration and seed (SC-07, NFR-AVAIL-001) |

**Startup sequence:**

```
dotnet run (Api project)
  тФФтФАтФА VacaFlowDbContext.MigrateAsync()       тЖТ Applies pending EF Core migrations
      тФФтФАтФА SeedDataInitializer.RunAsync()     тЖТ Seeds AbsenceTypes, Manager account, Employee-Manager assignment
          тФФтФАтФА ASP.NET Core Kestrel           тЖТ API ready, listening on configured port
```

All three steps execute automatically at startup. No manual database setup step is required (NFR-MAINT-004, NFR-COMPAT-001).

### 3.2 Frontend тАФ Next.js Dev Server

The Next.js frontend runs as the standard Next.js development server. No build-and-serve production configuration is required for the MVP.

| Attribute | Value |
|-----------|-------|
| Framework | Next.js 14 / React 18 / TypeScript |
| Process model | Single foreground `npm run dev` process |
| Default port | `http://localhost:3000` |
| Proxy configuration | `next.config.js` proxies all `/api/**` requests to the ASP.NET Core API (eliminates CORS configuration) |
| Hot reload | Enabled (standard Next.js dev server behavior) |
| Memory footprint (estimated) | 200тАУ500 MB RAM during active development |
| Build step | Not required for acceptance walkthrough; `npm run dev` is sufficient |

**Proxy rule (conceptual тАФ implemented in `next.config.js`):**

```
Incoming: GET http://localhost:3000/api/requests
Proxied:  GET http://localhost:5000/api/requests
```

This proxy eliminates the need for CORS headers or any network-level configuration (SAD-DD-001 ┬з1.1, ┬з2.2).

### 3.3 Process Concurrency Model

Both processes run independently on the local machine. They communicate via the localhost TCP loopback. No inter-process coordination infrastructure is required.

| Process | Port | Protocol | Direction |
|---------|------|----------|-----------|
| ASP.NET Core API (Kestrel) | 5000 (default) | HTTP | Receives from Next.js proxy and direct API clients |
| Next.js Dev Server | 3000 (default) | HTTP | Receives from browser; proxies `/api` to port 5000 |

## 4. Persistence Requirements

### 4.1 SQLite Database

VacaFlow_14 uses SQLite as its sole persistence store, managed via EF Core 8 Code-First (ADR-004 in SAD-DD-001 ┬з4). No external database server is required.

| Attribute | Value |
|-----------|-------|
| Engine | SQLite 3 (embedded via `Microsoft.EntityFrameworkCore.Sqlite`) |
| Provisioning | Created automatically by EF Core on first `dotnet run` if the file does not exist |
| Migration strategy | `MigrateAsync()` called at application startup тАФ idempotent, applies only pending migrations |
| File path | Configured in `appsettings.Development.json` (e.g., `vacaflow.db` in the API project root) |
| File size (estimated MVP) | < 10 MB for the full acceptance walkthrough dataset |
| Access mode | Single-writer; local loopback only |
| ACID compliance | Full for single-user local workloads (SQLite default WAL mode) |

> **Concurrency assumption:** SQLite is appropriate for the 1тАУ5 concurrent reviewer users assumed for this MVP. This assumption is **Not Validated** (SI-001 ┬з5 Assumptions, SI-001 ┬з6 open Critical Information Gap). Concurrent write throughput at scale is a known SQLite limitation (SAD-DD-001 ADR-004, Consequences тАФ Negative; SAD-001 ┬з9 R-02). If the user population grows beyond the reviewer team, a client-server database must be evaluated before production promotion.

### 4.2 Database Schema Overview

The schema is managed entirely by EF Core Code-First migrations. The four canonical entities align with the domain model defined in SAD-DD-001 ┬з3.1:

| Table | Key Columns | Notes |
|-------|-------------|-------|
| `Employees` | `Id`, `FullName`, `Email` (unique), `PasswordHash`, `Role` (Employee\|Manager), `ManagerId` (nullable FK тЖТ Employees) | `ManagerId` implements the Employee-to-Manager assignment (BR-DATA-002 / FR-AUTH-010). Open item R-01: assignment cardinality pending Sponsor confirmation (SAD-DD-001 ┬з1.1). |
| `AbsenceTypes` | `Id`, `Name` | Seeded catalog: Vacation, Personal Leave, Sick Leave. Not user-maintainable (FR-AUTH-008, BR-DATA-001). |
| `Requests` | `Id`, `OwnerEmployeeId` (FK), `AbsenceTypeId` (FK), `StartDate`, `EndDate`, `Reason`, `Status` | Implements the request lifecycle state machine (Draft тЖТ Submitted тЖТ Approved / Rejected / Cancelled). |
| `Approvals` | `Id`, `RequestId` (FK, unique), `DecidedByEmployeeId` (FK), `Decision`, `DecidedAt`, `Comment` (nullable) | Exactly one record per approval/rejection decision (NFR-REL-003). |

### 4.3 Seed Data Requirements

The `SeedDataInitializer` (SAD-DD-001 ┬з3.3.5) runs idempotently at startup and must guarantee:

| Seed Item | Requirement | Source |
|-----------|-------------|--------|
| AbsenceTypes | Exactly 3 records: Vacation, Personal Leave, Sick Leave | FR-AUTH-008, BR-DATA-001 |
| Manager account | At least 1 Employee with `Role = Manager`; credentials must use non-real placeholder values (e.g., `manager@example.com`) | FR-AUTH-007, NFR-SEC-004 |
| Employee-Manager assignment | Every non-manager Employee must have a valid `ManagerId` pointing to an existing Manager | FR-AUTH-010, BR-DATA-002, open item R-01 |
| Idempotency | Re-running on an existing database does not duplicate or overwrite seed data | NFR-REL-004 |

### 4.4 File Storage

VacaFlow_14 does not use object storage, CDN, or external file storage in the MVP. All application artifacts reside on the local developer workstation filesystem:

| Artifact | Location | Notes |
|----------|----------|-------|
| SQLite database file | API project root (path in `appsettings.Development.json`) | Must be listed in `.gitignore` (NFR-SEC-004) |
| Application source code | Git repository clone | Managed via Git |
| Frontend assets | Next.js `public/` and `src/` directories | Served by the Next.js dev server |
| EF Core migration files | `Infrastructure/Migrations/` (source-controlled) | Applied automatically at startup |
| Application logs | Console / standard output (see ┬з7) | Not persisted to a separate file in MVP |

> **TLS / HTTPS:** HTTPS is not required for localhost MVP operation. Kestrel's default HTTP configuration is sufficient. If the reviewer's browser enforces HTTPS for localhost (e.g., via HSTS preload), the Next.js dev server can be configured to proxy over HTTP to Kestrel without certificate management concerns.

### 5.3 CORS Configuration

CORS is not required because the Next.js dev server proxies all `/api/**` requests to the ASP.NET Core API on the same host. The browser sees all requests as same-origin (`localhost:3000`). The ASP.NET Core API does not need to configure a CORS policy for the MVP (SAD-DD-001 ┬з1.1, ADR-001 Consequences).

### 5.4 Anti-Forgery (CSRF) Token Flow

All state-changing requests (POST, PUT, PATCH, DELETE) are protected by ASP.NET Core's built-in anti-forgery middleware (ADR-003 in SAD-DD-001 ┬з4). The CSRF token flow operates over the loopback interface:

1. Browser loads the frontend page; the server sets an anti-forgery cookie.
2. Next.js frontend reads the cookie value and includes it as a request header on all state-changing API calls.
3. ASP.NET Core anti-forgery middleware validates the header against the cookie on every state-changing endpoint.

No additional network infrastructure is required for this flow.

## 6. Security Requirements

Security controls in the MVP context are constrained to what is enforceable in a local execution model. They are derived from NFR-001 v3.0 (NFR-SEC-001 through NFR-SEC-005) and SAD-DD-001 ┬з4 (ADR-003).

### 6.1 Authentication

| Control | Implementation | Requirement Source |
|---------|----------------|-------------------|
| Password hashing | BCrypt via `BCrypt.Net-Next`; applied in `RegisterSlice` before any database write; no plain-text storage at any point | NFR-SEC-001 (Critical) |
| Session management | ASP.NET Core Cookie Authentication; HTTP-only, SameSite=Lax cookies; session persisted server-side | NFR-SEC-005 (Critical) |
| Logout invalidation | `HttpContext.SignOutAsync()` invalidates the server-side session; subsequent requests with the old cookie return 401 | NFR-SEC-005 (Critical) |
| CSRF protection | Anti-forgery middleware applied globally on all state-changing endpoints (POST, PUT, PATCH, DELETE) | ADR-003 (SAD-DD-001) |
| Session across navigation | Cookie session survives page refresh and new-tab navigation without re-authentication | NFR-AVAIL-001 |

### 6.2 Authorization

| Control | Implementation | Requirement Source |
|---------|----------------|-------------------|
| Identity derivation | All endpoints read user identity from `HttpContext.User` (encrypted cookie); no trusted user identifier accepted in request body or query string | NFR-SEC-002 (Critical) |
| Role enforcement | Manager-only and Employee-only route protection enforced in slice handlers before domain logic; 403 returned on role mismatch | NFR-SEC-003 (Critical) |
| Ownership check | Request owner verified against authenticated user identity before edit, submit, or cancel operations; 403 on mismatch | NFR-SEC-003 (Critical) |
| Manager-assignment check | `Employee.ManagerId` verified in `ApproveRequestSlice` / `RejectRequestSlice`; manager may only act on requests belonging to their assigned employees тАФ **Note:** this check depends on the one-to-one Manager-to-Employee assignment model (BR-DATA-002 / FR-AUTH-010), which remains pending formal Sponsor confirmation (open item R-01, SAD-DD-001 ┬з1.1; NFR-SEC-003 note) | NFR-SEC-003 (Critical) |
| Self-approval prohibition | `ApproveRequestSlice` and `RejectRequestSlice` reject any request where the authenticated manager is the request owner; 403 returned | NFR-SEC-003 (Critical) |
| State-gate enforcement | `RequestStateMachine` (Domain layer) rejects invalid transitions; 422 returned via `GlobalExceptionHandler` | NFR-REL-001 (Critical) |

### 6.3 Data Protection

| Control | Implementation | Requirement Source |
|---------|----------------|-------------------|
| Database file excluded from source control | `*.db` and `*.sqlite` entries in `.gitignore`; no database file with real credentials committed | NFR-SEC-004 (Critical) |
| Seed credentials | All seed accounts use clearly non-real placeholder values (e.g., `manager@example.com`, `employee@example.com`) | NFR-SEC-004 (Critical) |
| Database file location | Documented in the project README so reviewers can locate and back up the file manually | NFR-SEC-004 (Critical), NFR-MAINT-004 |
| Application secrets | No production secrets in source control; `appsettings.Development.json` excluded from version control or contains only local development values | NFR-SEC-004 (Critical) |

### 6.4 Transport Security

| Control | Implementation | Notes |
|---------|----------------|-------|
| TLS | Not required for localhost loopback communication | All traffic remains on the local machine; no external network exposure |
| HTTPS | Optional for local development; Kestrel HTTP is sufficient for the reviewer walkthrough | If enforced, self-signed certificate may be used via `dotnet dev-certs` |
| Data in transit | All API communication travels over the localhost TCP loopback; no internet exposure | Acceptable for MVP local execution context |

### 6.5 Exception Handling and Error Disclosure

| Control | Implementation | Requirement Source |
|---------|----------------|-------------------|
| `GlobalExceptionHandler` | Catches all unhandled exceptions; returns RFC 7807 Problem Details responses; does not leak stack traces or internal implementation details to the client | NFR-AVAIL-001 (SC-08) |
| Error mapping | `DomainException` тЖТ 422, `ValidationException` тЖТ 400, authorization failure тЖТ 403, `NotFoundException` тЖТ 404, all others тЖТ 500 | SAD-DD-001 ┬з3.3.6 |

## 8. Backup and Recovery

### 8.1 Data Backup

No automated backup infrastructure is in scope for the MVP. The SQLite database file is the sole durable artifact. Manual backup procedures are sufficient:

| Procedure | Steps |
|-----------|-------|
| Manual file backup | Stop the API (`Ctrl+C`); copy the SQLite `.db` file to a safe location; restart the API |
| Restore from backup | Stop the API; replace the `.db` file with the backup copy; restart the API |
| File location | Documented in the project README (NFR-MAINT-004, NFR-SEC-004) |

### 8.2 Recovery Targets

No formal RPO or RTO is defined for the MVP. The following guidance applies to the acceptance walkthrough context:

| Scenario | Resolution |
|----------|-----------|
| Accidental data deletion | Restore from manual backup if taken; otherwise re-seed by deleting the `.db` file and restarting the API (triggers fresh migration and seeding) |
| Application crash | Restart the API (`dotnet run`); SQLite guarantees ACID durability тАФ no data loss for committed transactions |
| Corrupted database file | Delete the `.db` file and restart the API; data from the session is lost (reviewer must re-run the acceptance workflow) |

### 8.3 Schema Migration Recovery

| Scenario | Resolution |
|----------|-----------|
| Migration fails at startup | Review the EF Core error in the console log; common causes: schema conflict, duplicate migration. Run `dotnet ef database drop` (in development only) and restart to rebuild from scratch. |
| Schema rollback needed | EF Core supports `dotnet ef database update {PreviousMigration}` to roll back to a prior schema version |
