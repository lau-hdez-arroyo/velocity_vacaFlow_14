## 1. Defect Management Overview

### 1.1 Purpose

This document defines the defect management process and provides tracking templates for VacaFlow_14. Its primary goal is to ensure that blocking issues discovered during MVP validation are identified, recorded, and resolved in a systematic and transparent manner, protecting the integrity of the core vacation-request workflow before final acceptance.

### 1.2 Scope

This defect log covers:

- Defects discovered during MVP local-execution validation
- Issues affecting the core workflow: registration, login, vacation request creation, submission, approval, rejection, and final status visibility
- Cosmetic and non-blocking observations
- Identity and authorization violations (e.g., approvals performed by non-managers)

The following are **out of scope** for this log:

- Post-MVP production incidents (no broad deployment planned)
- Enhancement requests (tracked separately in the product backlog)
- CI/CD pipeline defects (no pipeline exists for this MVP)

### 1.3 Tracking Tool

**Primary Tool:** Azure DevOps (Work Items — Bug type)
**Reporting:** Azure DevOps Queries and Dashboards

### 1.4 Validation Context

VacaFlow_14 is delivered as source code for local execution. Validation is performed by James Parker (Operations Manager, project sponsor), who may involve one additional manager and one employee representative to exercise the full workflow. No formal production support or SLA framework applies beyond this MVP review window.

## 2. Defect Classification

### 2.1 Severity Levels

| Severity | Definition | Examples | Resolution Target |
|----------|------------|----------|-------------------|
| **Blocking** | Prevents any step of the core workflow from completing; the MVP cannot be accepted while this issue is open | Registration fails; login rejected for valid credentials; request creation errors out; submission does not reach manager; approval/rejection not persisted; final status not visible to employee; identity bypass allowing a non-manager to approve | Highest priority — addressed before next validation session |
| **Cosmetic** | Non-blocking; the core workflow completes correctly but a minor visual or UX imperfection is present | Label truncation; minor alignment drift; informational text wording; non-critical empty state | Addressed after all Blocking defects are closed; may be accepted with documented waiver |

### 2.2 Priority Levels

| Priority | Mapped To | Action |
|----------|-----------|--------|
| **P1 – Immediate** | Blocking | Fix before resuming validation; escalate to Emily Harrison and James Parker immediately |
| **P2 – Deferred** | Cosmetic | Log and schedule for resolution after all Blocking items are closed |

### 2.3 Defect Types

| Type | Description | Relevance to VacaFlow_14 |
|------|-------------|--------------------------|
| Functional | Feature does not behave as specified | Core workflow steps fail |
| Authorization | Identity or role enforcement violated | Non-manager can approve; authenticated identity bypassed |
| Data Integrity | Data not persisted, corrupted, or displayed incorrectly | Request status not saved or shown accurately |
| Usability | Workflow is confusing or difficult to complete correctly | Multi-step form issues, unclear labels |
| Compatibility | Local runtime environment dependency | OS/browser-specific failures during local execution |
| Regression | Previously passing step broken by a subsequent fix | Fix to one workflow step breaks another |

## 7. Defect Triage Process

### 7.1 Triage Cadence

Given the small, focused validation team for VacaFlow_14, triage is lightweight and synchronous:

| Aspect | Details |
|--------|---------|
| **Trigger** | Any new defect logged in Azure DevOps by a validation participant |
| **Triage Interval** | Within the same validation session when possible; within one business day otherwise |
| **Participants** | Emily Harrison (primary triage), James Parker (severity confirmation for Blocking items) |
| **Channel** | Direct message or shared Azure DevOps board review |
| **Duration** | 15 minutes per defect; 30 minutes for batch review |

### 7.2 Triage Checklist

- [ ] Is this a reproducible defect (not a one-time environment glitch)?
- [ ] Is this a duplicate of an existing open item?
- [ ] Does this affect any step of the core workflow (Blocking) or is it cosmetic only?
- [ ] Is there sufficient information to reproduce it (steps, environment, account)?
- [ ] Is an authorization or identity enforcement rule violated?
- [ ] Who on the development team should own this fix?
- [ ] Can a workaround allow validation to continue while the fix is prepared?

### 7.3 Triage Decisions

| Decision | Criteria | Next Step |
|----------|----------|-----------|
| Fix — Immediate | Blocking severity; prevents workflow completion or violates authorization | Assign to development team; halt further validation of affected workflow step |
| Fix — Deferred | Cosmetic severity; no workflow impact | Log with P2; address after all Blocking items closed |
| More Information Needed | Cannot reproduce; insufficient steps or environment detail | Return to reporter with specific questions |
| Reject — Duplicate | Same root cause as existing open defect | Close with link to original |
| Reject — Not a Defect | Behavior is consistent with documented requirements | Close with explanation and reference to requirement |
| Accept with Waiver | Cosmetic; James Parker accepts as-is for MVP | Close as Deferred with written acceptance recorded in Azure DevOps comment |
