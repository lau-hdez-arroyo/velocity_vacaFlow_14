## 1. Introduction

### 1.1 Purpose

This document establishes the coding standards and best practices for VacaFlow_14, a local-execution MVP for managing employee absence and vacation requests. It exists to ensure that all code produced for this project is correct, secure, readable, and consistent with the architectural decisions recorded in SAD-DD-001.

These standards are written for a single-developer project and are intentionally lean. They avoid ceremony that adds overhead without proportional quality benefit. Every rule here exists to protect one or more of the project's primary quality targets: **correctness, clarity, and reliability**.

### 1.2 Scope

These standards apply to all code written for VacaFlow_14:

- Backend: C# 12 / .NET 8 ASP.NET Core Minimal API, Entity Framework Core 8
- Frontend: TypeScript, React 18, Next.js 14
- Test code: xUnit (backend), Jest / React Testing Library (frontend)
- Configuration files and scripts

### 1.3 Guiding Principles

| Principle | Statement |
|-----------|-----------|
| **Correctness first** | Business rules enforced by the API, never delegated to the UI |
| **Security by default** | Passwords are always hashed; identity is always server-derived |
| **Clarity over cleverness** | Prefer readable, explicit code over concise abstractions |
| **No unnecessary complexity** | No MediatR, no CQRS, no generic repository patterns beyond project need |
| **Fail loudly** | Invalid operations return clear, structured error responses |

### 1.4 Architecture Alignment

All standards in this document are consistent with the Pragmatic Vertical Slice Onion architecture selected in SAD-DD-001 (ADR-001 through ADR-005). Key architectural constraints that inform these standards:

- Four layers: **Domain, Application, Infrastructure, API** — no MediatR, no messaging frameworks
- One file per operation (vertical slice): `CreateRequestSlice.cs`, `ApproveRequestSlice.cs`, etc.
- `RequestStateMachine` lives in the **Domain layer** as a static class with zero outward dependencies
- `ValidationPipelineInvoker` runs all `IValidator<T>` instances before any handler executes
- Authentication uses **ASP.NET Core Cookie Session** — no JWT for this MVP
- Identity is always read from `HttpContext.User` — never from the request body

### 2.4 Comments and Documentation

#### When to Comment

Comment when the **why** is not obvious from reading the code:

- Non-trivial business rule being enforced (e.g., "// BR-MGR-004: manager cannot approve their own request")
- Reference to an ADR or architecture decision (e.g., "// ADR-003: identity always read from cookie, never from body")
- Intentional workaround with a note about the limitation

#### When Not to Comment

- Obvious code that reads as plain English
- Version history — that belongs in Git
- Commented-out code — delete it; Git preserves history

#### Comment Style (C#)

```csharp
// Single-line for brief inline notes

// BR-APPR-001: the Approval record and the status transition must be
```

## 3. C# / ASP.NET Core Standards

### 3.1 Vertical Slice Structure

Each slice file contains exactly one HTTP operation: the endpoint registration, the request/response record types, the handler delegate or class, and the inline validator registration reference. This is the canonical pattern from SAD-DD-001.

```csharp
// CreateRequestSlice.cs — Application/Slices/Requests/
namespace VacaFlow.Application.Slices.Requests;

public static class CreateRequestSlice
{
    public record Command(
        Guid AbsenceTypeId,
        DateOnly StartDate,
        DateOnly EndDate,
        string Reason
    );

    public record Response(Guid Id, string Status);

    public static void Register(WebApplication app)
    {
        app.MapPost("/api/requests", HandleAsync)
           .RequireAuthorization();
    }

    private static async Task<IResult> HandleAsync(
        Command command,
        HttpContext httpContext,
        VacaFlowDbContext db,
        ValidationPipelineInvoker validator,
        CancellationToken cancellationToken)
    {
        // Step 1: Validate
        var validation = await validator.InvokeAsync(command);
        if (!validation.IsValid)
            return Results.ValidationProblem(validation.Errors);

        // Step 2: Derive identity from session — never from command
        // ADR-003: identity comes from the authenticated cookie session
        var employeeId = httpContext.User.GetEmployeeId();

        // Step 3: Create domain entity in Draft state
        var request = new AbsenceRequest(employeeId, command.AbsenceTypeId,
            command.StartDate, command.EndDate, command.Reason);

        db.Requests.Add(request);
        await db.SaveChangesAsync(cancellationToken);

        return Results.Created($"/api/requests/{request.Id}",
            new Response(request.Id, request.Status.ToString()));
    }
}
```

**Rules for slice files:**

1. One slice file per HTTP operation — no combining Create + Edit into one file
2. The handler reads caller identity from `HttpContext.User`, never from the command payload
3. The validation pipeline is invoked before any domain logic or database access
4. A single `SaveChangesAsync` call commits all changes (Unit of Work, NFR-REL-003)
5. Slices depend inward: they may reference Domain types and Infrastructure interfaces, but Domain types must not reference Application or Infrastructure

### 3.2 Domain Layer Rules

The Domain layer contains `RequestStateMachine`, `DomainException`, all entity classes, enums, and the `IPasswordHasher` port. **No EF Core attribute, no ASP.NET Core reference, and no NuGet package reference may appear inside the Domain layer.**

```csharp
// Domain/RequestStateMachine.cs — ZERO external dependencies
namespace VacaFlow.Domain;

public static class RequestStateMachine
{
    private static readonly Dictionary<(RequestStatus, RequestAction), RequestStatus> _transitions
        = new()
        {
            [(RequestStatus.Draft,     RequestAction.Submit)]  = RequestStatus.Submitted,
            [(RequestStatus.Draft,     RequestAction.Cancel)]  = RequestStatus.Cancelled,
            [(RequestStatus.Submitted, RequestAction.Approve)] = RequestStatus.Approved,
            [(RequestStatus.Submitted, RequestAction.Reject)]  = RequestStatus.Rejected,
            [(RequestStatus.Submitted, RequestAction.Cancel)]  = RequestStatus.Cancelled,
        };

    /// <summary>Applies the action to the request. Throws DomainException on invalid transitions.</summary>
    public static void Transition(AbsenceRequest request, RequestAction action)
    {
        var key = (request.Status, action);
        if (!_transitions.TryGetValue(key, out var nextStatus))
            throw new DomainException(
                $"Cannot apply '{action}' to a request in '{request.Status}' status.");

        request.Status = nextStatus;
    }

    /// <summary>Returns the valid actions for the given status. Used by the API to drive UI hints.</summary>
    public static IReadOnlyList<RequestAction> GetValidActions(RequestStatus status)
        => _transitions.Keys
                       .Where(k => k.Item1 == status)
                       .Select(k => k.Item2)
                       .ToList();
}
```

**Domain layer checklist:**

- [ ] No `using Microsoft.EntityFrameworkCore` anywhere in the Domain project
- [ ] No `using Microsoft.AspNetCore` anywhere in the Domain project
- [ ] Entity classes have no EF Core data annotation attributes (`[Key]`, `[Column]`, etc.) — all mappings go in `VacaFlowDbContext` via Fluent API
- [ ] `DomainException` messages are human-readable (they surface to the API caller via `GlobalExceptionHandler`)

### 3.3 Entity Classes

```csharp
// Domain/Entities/AbsenceRequest.cs
namespace VacaFlow.Domain.Entities;

public class AbsenceRequest
{
    // Private setters — state can only change through domain methods
    public Guid Id { get; private set; } = Guid.NewGuid();
    public Guid OwnerEmployeeId { get; private set; }
    public Guid AbsenceTypeId { get; private set; }
    public DateOnly StartDate { get; private set; }
    public DateOnly EndDate { get; private set; }
    public string Reason { get; private set; }
    public RequestStatus Status { get; internal set; } = RequestStatus.Draft;

    // Required by EF Core
    private AbsenceRequest() { Reason = string.Empty; }

    public AbsenceRequest(
        Guid ownerEmployeeId,
        Guid absenceTypeId,
        DateOnly startDate,
        DateOnly endDate,
        string reason)
    {
        OwnerEmployeeId = ownerEmployeeId;
        AbsenceTypeId = absenceTypeId;
        StartDate = startDate;
        EndDate = endDate;
        Reason = reason;
    }

```

**Note:** `Status` has `internal set` rather than `private set` so that `RequestStateMachine` (in the same Domain assembly) can apply transitions. No external assembly may set `Status` directly.

### 3.4 Async and Cancellation

```csharp
// ✓ Correct — async all the way through; cancellationToken propagated
private static async Task<IResult> HandleAsync(
    Command command,
    VacaFlowDbContext db,
    CancellationToken cancellationToken)
{
    var request = await db.Requests
        .FirstOrDefaultAsync(r => r.Id == command.Id, cancellationToken);

    if (request is null)
        return Results.NotFound();

    await db.SaveChangesAsync(cancellationToken);
    return Results.Ok();
}

// ✗ Incorrect — blocking on async, no cancellation support
private static IResult Handle(Command command, VacaFlowDbContext db)
{
    var request = db.Requests.FirstOrDefault(r => r.Id == command.Id);
    db.SaveChanges();
    return Results.Ok();
}
```

**Rules:**

1. All database calls use the `Async` variants (`FirstOrDefaultAsync`, `SaveChangesAsync`, `ToListAsync`)
2. Every `async` method accepts and propagates `CancellationToken`
3. Never use `.Result` or `.Wait()` on a `Task` — it causes deadlocks under ASP.NET Core

### 3.5 Null Safety

```csharp
// ✓ Correct — enable nullable in all projects
#nullable enable

// ✓ Correct — null-conditional and pattern matching
if (await db.Employees.FirstOrDefaultAsync(e => e.Id == id) is not { } employee)
    return Results.NotFound();

// ✓ Correct — null-coalescing for fallback values
var comment = approvalCommand.Comment ?? string.Empty;

// ✗ Incorrect — unchecked null dereference
var employee = db.Employees.FirstOrDefault(e => e.Id == id);
var name = employee.FullName; // NullReferenceException if not found
```

**Rule:** Enable `<Nullable>enable</Nullable>` in every project's `.csproj`. Treat all compiler nullable warnings as errors in the Domain project.

### 3.6 EF Core Query Rules

```csharp
// ✓ Correct — project to a DTO immediately; do not load the full entity when summary is enough
var summaries = await db.Requests
    .Where(r => r.OwnerEmployeeId == employeeId)
    .OrderByDescending(r => r.StartDate)
    .Select(r => new RequestSummary(r.Id, r.Status, r.StartDate, r.EndDate))
    .ToListAsync(cancellationToken);

// ✓ Correct — eager-load related data when needed to avoid N+1
var request = await db.Requests
    .Include(r => r.AbsenceType)
    .Include(r => r.Approval)
    .FirstOrDefaultAsync(r => r.Id == id, cancellationToken);

// ✗ Incorrect — N+1: loads every request then hits DB for each AbsenceType
foreach (var r in await db.Requests.ToListAsync())
{
    var typeName = r.AbsenceType.Name; // lazy load per row
}
```

**Rules:**

1. Never access navigation properties without `Include` or explicit projection — no lazy loading is configured
2. Always filter (`Where`) before projecting (`Select`) to avoid loading unused data
3. Use `.Select()` projections to DTOs for list endpoints; use `.Include()` for detail endpoints that need the full object graph

Register slice endpoints and validators in `Program.cs`. The registration is explicit and intentional — no auto-discovery scanning.

### 4.3 Shared Domain Types

```typescript
// src/types/index.ts — single source of truth for shapes shared with the API

export type RequestStatus = 'Draft' | 'Submitted' | 'Approved' | 'Rejected' | 'Cancelled';
export type RequestAction = 'Submit' | 'Cancel' | 'Approve' | 'Reject';
```

### 4.4 React Component Standards

```tsx
// ✓ Correct — functional component with explicit prop interface
// src/components/RequestCard.tsx

interface RequestCardProps {
  request: AbsenceRequest;
  onSubmit?: (id: string) => void;
  onCancel?: (id: string) => void;

export function RequestCard({ request, onSubmit, onCancel }: RequestCardProps) {
```

**Rules:**

1. Every component has an explicit props interface above the component definition
2. Action buttons are gated by `validActions` from the API response — no local status-to-action mapping in the frontend
3. Approver and owner identities are never passed from the frontend to the backend for business decisions — the backend always derives them from `HttpContext.User`
4. No class components

### 4.5 React Hooks Rules

```typescript
// ✓ Correct — data fetching with proper error and loading state
function useRequestDetail(id: string) {
  const [data, setData] = useState<AbsenceRequest | null>(null);
  const [error, setError] = useState<string | null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    let cancelled = false;
    fetchRequestDetail(id)
      .then(r => { if (!cancelled) setData(r); })
      .catch(e => { if (!cancelled) setError(e.message); })
      .finally(() => { if (!cancelled) setLoading(false); });
    return () => { cancelled = true; };
  }, [id]);

  return { data, error, loading };
}
```

**Rules:**

1. Custom hooks are named with the `use` prefix
2. All `useEffect` hooks that trigger async operations must handle the cleanup/cancellation case (set `cancelled = true` on unmount)
3. Do not call hooks conditionally

## 5. Security Standards

Security is a non-negotiable constraint for VacaFlow_14. The following rules enforce the security properties validated in the architecture evaluation (AED-001, SC-01 through SC-03) and the NFRs in NFR-001 v3.0.

### 5.1 Password Handling (NFR-SEC-001 / SC-01)

```csharp
// ✓ Correct — always hash via the IPasswordHasher port
// Infrastructure/Security/BCryptHasher.cs
public class BCryptHasher : IPasswordHasher
{
    // Work factor 12 balances security and acceptable login latency for a single developer
    private const int WorkFactor = 12;

    public string Hash(string plainTextPassword)
        => BCrypt.Net.BCrypt.HashPassword(plainTextPassword, WorkFactor);

    public bool Verify(string plainTextPassword, string hashedPassword)
        => BCrypt.Net.BCrypt.Verify(plainTextPassword, hashedPassword);
}

// ✗ PROHIBITED — storing or logging plain-text passwords at any point
employee.PasswordHash = command.Password;  // Never
_logger.LogInformation("User registered with password {P}", command.Password); // Never
```

**Rules:**

1. The plain-text password must never be stored, logged, or included in any response
2. `BCryptHasher` is the only implementation of `IPasswordHasher`; no slice may call BCrypt directly
3. Password comparison always uses `BCryptHasher.Verify` — never string equality

### 5.2 Identity — Always Server-Derived (NFR-SEC-002 / SC-02)

```csharp
// ✓ Correct — identity read from the authenticated cookie session
var employeeId = httpContext.User.GetEmployeeId();
// GetEmployeeId() is an extension method on ClaimsPrincipal that reads the NameIdentifier claim

// ✗ PROHIBITED — reading identity from the request body or query string
var employeeId = command.EmployeeId;        // Never
var employeeId = httpContext.Request.Query["userId"]; // Never
```

**Rule:** No slice handler may accept an `EmployeeId`, `ApproverId`, or any user identity value from the client. The authenticated cookie is the only authoritative source of caller identity.

**TypeScript equivalent:**

```typescript
// ✓ Correct — frontend never includes identity in approval payloads
async function approveRequest(requestId: string, comment: string): Promise<void> {
  await fetch(`/api/requests/${requestId}/approve`, {
    method: 'POST',
    body: JSON.stringify({ comment }),  // Only the decision data; identity is derived server-side
    headers: { 'Content-Type': 'application/json' },
  });
}

// ✗ PROHIBITED — do not include managerId or userId from frontend state
await fetch(`/api/requests/${requestId}/approve`, {
  body: JSON.stringify({ comment, managerId: currentUser.id }), // Never
});
```

### 5.3 Authorization Checks Order in Approval Slices (NFR-SEC-003 / SC-03)

All authorization checks in `ApproveRequestSlice` and `RejectRequestSlice` must execute in this order, before any domain logic:

```csharp
// ApproveRequestSlice.cs — authorization guard order is mandatory
private static async Task<IResult> HandleAsync(
    ApproveCommand command,
    HttpContext httpContext,
    VacaFlowDbContext db,
    CancellationToken ct)
{
    // 1. Derive manager identity from session
    var managerId = httpContext.User.GetEmployeeId();

    // 2. Role check — BR-MGR-002 / NFR-SEC-003
    var manager = await db.Employees.FirstOrDefaultAsync(e => e.Id == managerId, ct);
    if (manager is null || manager.Role != Role.Manager)
        return Results.Forbid();

    // 3. Load the request
    var request = await db.Requests
        .Include(r => r.Owner)
        .FirstOrDefaultAsync(r => r.Id == command.RequestId, ct);
    if (request is null)
        return Results.NotFound();

    // 4. Manager-assignment check — FR-APPR-008 / BR-DATA-002
    if (request.Owner.ManagerId != managerId)
        return Results.Forbid();

    // 5. Self-approval prohibition — FR-APPR-009 / BR-MGR-004
    if (request.OwnerEmployeeId == managerId)
        return Results.Forbid();

    // 6. Domain state machine transition — FR-APPR-007
    RequestStateMachine.Transition(request, RequestAction.Approve);

    // 7. Persist atomically — NFR-REL-003 / BR-APPR-001
    db.Approvals.Add(new Approval(request.Id, managerId, Decision.Approved, command.Comment));
    await db.SaveChangesAsync(ct);

    return Results.Ok();
}
```

**Rule:** This check order is not optional. Role, assignment, and self-approval checks must always precede state machine invocation.

### 5.4 State Machine — Protect Submitted-State Edits (NFR-REL-001)

```csharp
// EditRequestSlice.cs — ownership + status guard before editing
if (request.OwnerEmployeeId != employeeId)
    return Results.Forbid();

// BR-REQ-003: only Draft requests may be edited
if (request.Status != RequestStatus.Draft)
    return Results.Problem(
        title: "Request cannot be edited",
        detail: $"Only Draft requests can be edited. This request is currently '{request.Status}'.",
        statusCode: 422);
```

**Rule:** The check that a request is in `Draft` before edit is enforced at the slice level (before calling the state machine) because `Edit` is not a state machine transition — it is a field update restricted to a specific state.

### 5.5 Anti-Forgery (CSRF) Compliance

The `UseAntiforgery()` middleware is registered globally in `Program.cs`. No additional per-endpoint attribute is required; all state-changing requests (`POST`, `PUT`, `DELETE`) require a valid XSRF token.

The frontend must include the anti-forgery token on all mutation requests. The token is obtained from the `/_antiforgery/token` endpoint after login.

## 7. Error Handling Standards

### 7.1 Exception Hierarchy

```
Exception
└── AppException (base for all application-controlled exceptions)
    ├── DomainException (422 — invalid domain operation, human-readable)
    ├── NotFoundException (404)
    ├── AuthorizationException (403)
    └── ValidationException (400 — aggregated field errors from pipeline)
```

No other exception types are defined for VacaFlow_14. Infrastructure exceptions (EF Core, IO) are caught by `GlobalExceptionHandler` and mapped to 500.

### 7.2 GlobalExceptionHandler Mapping

```csharp
// Middleware/GlobalExceptionHandler.cs
public class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
        => _logger = logger;

    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext,
        Exception exception,
        CancellationToken cancellationToken)
    {
        var (statusCode, title) = exception switch
        {
            DomainException      => (422, "Business Rule Violation"),
            ValidationException  => (400, "Validation Failed"),
            AuthorizationException => (403, "Forbidden"),
            NotFoundException    => (404, "Not Found"),
            _                    => (500, "Internal Server Error"),
        };

        if (statusCode == 500)
            _logger.LogError(exception, "Unhandled exception");

        httpContext.Response.StatusCode = statusCode;
        await httpContext.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Title = title,
            Detail = statusCode < 500 ? exception.Message : "An unexpected error occurred.",
            Status = statusCode,
        }, cancellationToken);

        return true;
    }
}
```

**Rules:**

1. Never let a raw exception propagate to the client — every exception class has a defined mapping
2. For 500 errors, the `Detail` field must say "An unexpected error occurred." — never leak the stack trace or internal error message
3. For 422 errors, `Detail` must contain the `DomainException.Message` (these messages are human-readable by design, as per §3.2)

### 7.3 Validation Error Response Shape

The `ValidationPipelineInvoker` returns errors in this shape, compatible with ASP.NET Core's `ValidationProblem`:

```json
{
  "title": "One or more validation errors occurred.",
  "status": 400,
  "errors": {
    "startDate": ["Start date must not be in the past."],
    "reason": ["Reason is required and must be between 1 and 500 characters."]
  }
}
```

**Rule:** Each field-level error message must be a complete, human-readable sentence. No error codes, no raw enum names, no technical jargon in validation messages.

### 7.4 Frontend Error Handling

```typescript
// lib/api-client.ts
export class ApiError extends Error {
  constructor(
    public readonly status: number,
    public readonly title: string,
    public readonly detail?: string,
    public readonly fieldErrors?: Record<string, string[]>
  ) {
    super(title);
  }
}

async function apiPost<T>(path: string, body: unknown): Promise<T> {
  const res = await fetch(path, {
    method: 'POST',
    body: JSON.stringify(body),
    headers: { 'Content-Type': 'application/json' },
  });

  if (!res.ok) {
    const problem = await res.json();
    throw new ApiError(res.status, problem.title, problem.detail, problem.errors);
  }

  return res.json() as Promise<T>;
}
```

**Rules:**

1. Always check `res.ok` before reading the response body as data
2. Propagate `ApiError` with the structured `fieldErrors` so form components can display per-field messages
3. Never swallow API errors silently — at minimum, log them to the console and show a user-visible message
