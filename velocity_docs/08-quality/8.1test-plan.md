## 1. Introduction

### 1.1 Purpose

This test plan defines the testing strategy, scope, schedule, resource requirements, and quality criteria for VacaFlow_14 — an internal absence and vacation request management system developed for IGS Solutions. VacaFlow replaces an informal approval process conducted via email and Microsoft Teams chat that produced a documented incident where an informal reply was treated as a formal approval without any recorded decision. Testing activities described in this document are designed to verify that the system correctly enforces the request lifecycle, role-based authorization, session-based identity derivation, and approval record integrity as specified in FRS-001 (v1.2) and NFR-001 (v3.0), and that it satisfies the acceptance criteria defined in the project governance framework.

### 1.2 Scope

#### In Scope

- **Authentication and session management:** User registration, login, logout, and current-user endpoint behavior (FR-AUTH-001 through FR-AUTH-010)
- **Employee request lifecycle:** Creating, editing, submitting, and cancelling absence requests; date validation; read-only enforcement on non-Draft states (FR-EMP-001 through FR-EMP-008)
- **Manager request lifecycle (as requester):** Manager creating and managing their own requests under employee-side rules (FR-MGR-REQ-001, FR-MGR-REQ-002)
- **Manager approval and rejection:** Review list filtering, approval, rejection, optional comment, Approval record integrity, and self-approval prohibition (FR-APPR-001 through FR-APPR-009)
- **Business rule enforcement at the API layer:** All 21 business rules (BR-USER-001 through BR-USER-003, BR-REQ-001 through BR-REQ-004, BR-MGR-001 through BR-MGR-002, BR-APPR-001 through BR-APPR-004, BR-STATE-001, BR-SEC-001 through BR-SEC-003, BR-FIELD-001, BR-DATA-001 through BR-DATA-002) as catalogued in the Business Rules document (BR-001 v1.2); cross-referenced to FRS-001 BRs via the equivalence table in BR-001 §11
- **Non-functional requirements:** Security constraints (NFR-SEC-001 through NFR-SEC-005), reliability and state machine (NFR-REL-001 through NFR-REL-004), usability baseline (NFR-USE-001, NFR-USE-002, NFR-USE-003), local performance (NFR-PERF-001), and local availability (NFR-AVAIL-001, NFR-AVAIL-002)
- **Reason field validation (BR-FIELD-001):** The reason field is required and must not exceed 500 characters; enforced at the API layer on creation and Draft edit
- **Seeded data validation:** Automatic database migration, seeded Manager account, seeded absence type catalog (FR-AUTH-007, FR-AUTH-008, FR-AUTH-009)
- **End-to-end acceptance workflow automation:** Full Playwright E2E suite covering the complete acceptance script defined in governance-doc.md §6.2 (GOV-001), authored and maintained by Laura Hernandez

### 2.1 Testing Levels

| Level | Description | Responsibility | Automation |
|-------|-------------|----------------|------------|

| Security Testing | Session identity enforcement, password storage, role bypass attempts via direct API calls | Laura Hernandez | Playwright for session/identity checks; Postman/curl for crafted bypass payloads |

### 2.2 Test Types

#### Functional Testing

- **Approach:** Black-box testing against requirements specified in FRS-001; each functional requirement has at least one positive and one negative test case; test case IDs follow the RTM-001 v5.0 convention (TC-AUTH-*, TC-EMP-*, TC-MGR-*, TC-APPR-*, TC-SEC-*, TC-E2E-*)
- **Coverage Target:** 100% of Must Have requirements (FR-AUTH-001 through FR-APPR-009, FR-AUTH-010)
- **Priority:** All FRs tagged Must Have are acceptance-gating; Should Have FRs are tested but defects may be deferred
- **Automation:** Integration tests cover API happy paths and primary error paths; Playwright E2E suite covers the full UI acceptance workflow; direct API tests cover negative and security paths

#### Reason Field Validation Testing (BR-FIELD-001)

- **Requirement:** The reason field on absence requests is mandatory and must not exceed 500 characters (BR-FIELD-001); enforced by the API on creation and on Draft edit
- **Approach:** Boundary value analysis on reason field character count; API-level negative path testing
- **Test Cases (TC-FIELD-*):**
  - TC-FIELD-001: Submit request with missing reason — expect 400/422 validation error
  - TC-FIELD-002: Submit request with empty string reason — expect 400/422 validation error
  - TC-FIELD-003: Submit request with reason of exactly 500 characters — expect success
  - TC-FIELD-004: Submit request with reason of 501 characters — expect 400/422 validation error
  - TC-FIELD-005: Edit Draft request to clear reason — expect 400/422 validation error
  - TC-FIELD-006: Edit Draft request with reason exceeding 500 characters — expect 400/422 validation error
- **Applied On:** Request creation (FR-EMP-001) and Draft edit (FR-EMP-002); also Manager as requester (FR-MGR-REQ-001)
- **Technique:** Boundary value analysis; API-level direct testing

#### State Machine Testing

- **Approach:** Exhaustive transition table testing — every valid transition is exercised and every prohibited transition is attempted and confirmed to be rejected
- **Coverage:** All transitions defined in NFR-REL-001 (5 valid, 6 invalid)
- **Technique:** State transition testing; each invalid path attempted directly via API to confirm 400/422 responses

#### Date Validation Testing

- **Approach:** Boundary value analysis on start date (today, yesterday, tomorrow) and end date (before start, equal to start, after start)
- **Coverage:** NFR-REL-002; BR-REQ-001 (end date before start date), BR-REQ-002 (start date in the past); applied on both request creation and Draft edit
- **Technique:** Equivalence partitioning and boundary value analysis

#### Authorization and Security Testing

- **Approach:** Direct API calls (Postman/curl) as unauthorized actors attempting to bypass role checks, forge identities, and perform prohibited operations; Playwright tests validate UI-layer session enforcement
- **Coverage:** NFR-SEC-002, NFR-SEC-003; all six authorization rules in the NFR-SEC-003 acceptance criteria table
- **Technique:** Negative path testing; crafted HTTP requests with substituted identifiers, wrong roles, and missing sessions

#### Regression Testing

- **Approach:** Re-execution of the full Playwright E2E suite after any defect fix; manual walkthrough of blocking-defect scenarios
- **Scope:** Core workflow: register → create Draft → edit → submit → cancel second Draft → manager approve/reject → view final result (GOV-001 §6.2)
- **Trigger:** After each blocking defect is resolved; prior to UAT sign-off

#### Password Storage Verification

- **Approach:** Post-registration SQLite database file inspection and code review
- **Coverage:** NFR-SEC-001; BR-SEC-001
- **Technique:** Visual inspection of stored password column; code review confirming BCrypt or equivalent hashing call at registration

#### Usability Baseline Testing

- **Approach:** Manual walkthrough of all role-and-state combinations; keyboard-only navigation; Playwright assertions on element visibility per state
- **Coverage:** NFR-USE-001 (no invalid action buttons), NFR-USE-002 (readable labels), NFR-USE-003 (keyboard navigation baseline)
- **Technique:** Exploratory review during system testing cycles; Playwright checks for button presence/absence per lifecycle state

### 2.3 Test Design Techniques

| Technique | Applied To |
|-----------|------------|
| Equivalence Partitioning | Date field validation; role assignment; reason field length |
| Boundary Value Analysis | Start date (today/yesterday/tomorrow); end date relative to start date; reason field (0, 500, 501 characters) |
| State Transition Testing | Request lifecycle state machine (all valid and invalid transitions) |
| Decision Table Testing | Authorization rules (role × ownership × state combinations) |
| Use Case Testing | UC-001 through UC-013 from FRS-001 |
| Negative Path Testing | All prohibited operations; API bypass attempts; forged identity payloads |

| Aspect | Approach |
|--------|----------|
