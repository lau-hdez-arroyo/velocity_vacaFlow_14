## 4. Risk Log

| ID | Risk | Prob. | Impact | Mitigation | Owner |
|----|------|-------|--------|------------|-------|
| R-01 | Scope creep toward full HR platform — uncontrolled additions beyond the four confirmed entities and five request states | High | High | Enforce the signed Won't v1 list (BLG-001 §3). Emily Harrison logs every scope change request in the Change Log before it is discussed. No scope item enters Sprint 1 without explicit written approval from James Parker that includes identification of what is removed to offset it. Any addition that cannot be offset by a Won't v1 removal is declined for this sprint. | James Parker |
| R-02 | Authentication implemented in a way that still allows the frontend to send a trusted employee or approver identifier for business decisions | Medium | High | Every slice handler reads identity exclusively from `HttpContext.User` (the ASP.NET Core cookie session). No request body or query parameter carrying an employee ID or approver ID is accepted for any business decision. This is validated by the bypass scenarios in BLG-001 acceptance criteria (items 4, 5, 8, 12, 14) during UAT. If a bypass succeeds during UAT, the affected slice is corrected before acceptance sign-off proceeds. | Emily Harrison |
| R-03 | Architectural complexity added beyond MVP needs — MediatR, CQRS, generic repositories, or messaging frameworks introduced during implementation | Medium | High | Alternative C (SAD-DD-001 ADR-001) prohibits MediatR (NFR-MAINT-002). If a prohibited pattern is found, the slice is refactored to the vertical-slice model before implementation continues on the next story. | Emily Harrison |

### Hard Constraints

- **Local execution only:** No Azure deployment, no Docker, no CI/CD pipeline. The MVP runs from source code on a local machine using `dotnet run` + `npm run dev` exclusively.
- **SQLite only:** SQLite is the only permitted database for the MVP. No cloud or external database server may be introduced.
- **Application-managed authentication:** Registration and login use application-managed local accounts with BCrypt-hashed passwords. Microsoft Entra ID and all corporate SSO providers are excluded from this MVP.
- **Session-derived identity:** The API must always derive the current user identity from the authenticated session cookie. The frontend must never send a trusted employee or approver identifier for any business decision. This constraint is a hard security requirement enforced by every slice handler.
- **Passwords hashed always:** Passwords must never be stored in plain text. BCrypt hashing is mandatory for all accounts, including seeded accounts.
- **Database file not publicly exposed:** The SQLite `.db` file must not be committed to source control and must not contain real credentials. Excluded via `.gitignore`.
- **Four entities, five states only:** Scope is strictly limited to Employee, Absence Type, Request, and Approval entities, and the five request states (Draft, Submitted, Approved, Rejected, Cancelled). No additional entities may be introduced without a formal scope change.
- **No prohibited screens:** No HR administration screens, no role management UI, and no account administration screens are permitted in this MVP.
- **API-enforced business rules:** All business rules — date validation, state transitions, ownership checks, manager authorization — must be enforced by the API, not only by the UI.
- **No prohibited architectural patterns:** MediatR, CQRS, generic repositories, and messaging frameworks are excluded. Implementation follows the vertical slice model defined in SAD-DD-001 (Alternative C, ADR-001, NFR-MAINT-002).
