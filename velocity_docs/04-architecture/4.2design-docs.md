## 1. Executive Summary

### 1.1 Purpose

This document describes the technical architecture for VacaFlow_14 (IGS Solutions), providing a consolidated reference of requirements, constraints, quality attributes, and architecture decisions to guide the development team. It integrates the functional specification (FRS-001), the non-functional requirements specification (NFR-001 v3.0), the requirements traceability matrix (RTM-001 v5.0), and the Utility Tree (signed 2026-07-21) into a single architecture communication document for all stakeholders.

### 1.2 Scope

This architecture covers the VacaFlow MVP application in its entirety, including: the Next.js/React web frontend, the ASP.NET Core Minimal API backend, SQLite persistence via Entity Framework Core, and application-managed authentication (registration, login, logout, current user). The architecture encompasses four business domains — Employee, Absence Type, Request, and Approval — and the full request lifecycle (Draft, Submitted, Approved, Rejected, Cancelled). The layered structure spans Domain, Application, Infrastructure, and Api layers following a reduced Onion Architecture.

**Explicitly excluded from this architecture:** Microsoft Entra ID and any corporate SSO provider, Azure deployment, Docker and CI/CD pipelines, email or Teams notifications, password reset and email verification flows, account administration screens, vacation balance and holiday calendar calculations, overlapping request validation, attachments, reporting or dashboards, HR views, multi-level approvals, approval delegation, payroll/HR/calendar/directory integrations, advanced audit logs, and automated backup mechanisms.

### 1.4 Architecture Summary

VacaFlow_14 adopts a **Reduced Onion Architecture** pattern (Domain, Application, Infrastructure, Api, Web layers), selected via Quality Attribute Weighting analysis with a weighted score derived from the Utility Tree evaluation. Key drivers: **Security** (weight 10, highest priority — password hashing and server-side identity derivation are hard acceptance-rejection conditions), **Reliability** (weight 9 — state machine integrity and date validation enforced at the API layer), and **Availability** (weight 7 — zero-intervention cold start from source code). Trade-offs accepted: Performance is scoped to single-user local SQLite conditions with no formal load testing; Compatibility excludes Docker, cloud services, and all browsers beyond the last two releases of the four major desktop browsers; Usability is addressed at a level appropriate for an internal MVP tool without formal WCAG certification.

## 3. Constraints and Assumptions

### 3.1 Constraints

> Source: `sources/01-intake-strategic-intake-sources.md` — Questions 11a, 11b, 11c

#### 3.1.1 Business Constraints

- The application must be demonstrable as a complete, working MVP within the agreed review window; any blocking defect found during the review period must be resolved before final acceptance sign-off.
- The scope is strictly limited to the request lifecycle defined in the functional specification; no optional capabilities, speculative features, or advanced HR platform functions may be added.
- At least one Manager account and the three standard absence types (Vacation, Personal Leave, Sick Leave) must be available from a cold start without any manual data entry, so that a reviewer can execute the full acceptance workflow immediately.
- The catalog of absence types is fixed for the MVP; no catalog maintenance screen is included.
- Manager assignment is established at registration time and does not change during the MVP lifecycle; no reassignment or delegation flow is needed.

#### 3.1.2 Legal Constraints

- No formal regulatory compliance framework (GDPR, HIPAA, SOC 2) is required for the MVP. If VacaFlow is promoted to a production system, privacy policy, data retention rules, and formal compliance requirements must be assessed before promotion.
- The SQLite database file must not be committed to source control with real credentials or personal data. Seeded accounts must use clearly non-real credential values.
- No privacy notice or consent flow is required in the MVP application; the sponsor acknowledgment is recorded in SI-001 §5 Assumptions.

#### 3.1.3 Technical Constraints

- The application must run entirely from source code on a local machine using only `dotnet run` (API) and `npm run dev` (frontend), without Docker, Azure, any cloud service, CI/CD pipeline, or external database server.
- The backend stack is constrained to ASP.NET Core Minimal API and Entity Framework Core with SQLite. No server-side ORM other than EF Core is permitted.
- The frontend stack is constrained to Next.js and React. The frontend communicates with the API exclusively over HTTP.
- Architecture must follow a reduced Onion Architecture (Domain, Application, Infrastructure, Api, Web layers). No MediatR, generic repository abstractions, event bus frameworks, or CQRS segregation frameworks are permitted.
- Passwords must be stored as cryptographic hashes (BCrypt or equivalent); plain-text password storage is a hard rejection condition.
- The API must always derive the current user identity and the responsible approver from the authenticated server-side session or token. The frontend must never send a trusted employee or approver identifier as a business parameter.
- Microsoft Entra ID, corporate SSO, email notifications, Teams notifications, password reset, email verification, Docker, and CI/CD pipelines are explicitly excluded from the MVP.
- Automatic EF Core database migration must run on startup without manual intervention.

### 3.2 Assumptions

- The application will be run locally from source code only; no network deployment, cloud hosting, or shared server environment is assumed for the MVP.
- A single SQLite database file is sufficient for the MVP's data volume and concurrent usage, given that only a small number of reviewers and test accounts will use the system during the validation window. *(Note: the 1–5 concurrent user estimate originates from SI-001 §5 Assumptions, where it is marked as Not Validated, and remains an open Critical Information Gap per SI-001 §6.)*
- At least one manager account will be available either through seed data or through the registration flow before approval scenarios are tested.
- The absence type catalog (Vacation, Personal Leave, Sick Leave) is fixed for the MVP and does not require a maintenance screen.
- Manager assignment is established at registration time and does not change during the MVP lifecycle; no reassignment or delegation flow is needed.
- Password hashing is the only credential security requirement for the MVP; multi-factor authentication, password reset, and email verification are not required.
- The implementation team may choose the simplest consistent session or token mechanism (cookie or JWT) for maintaining authenticated state, provided that the server always derives user identity from it and the frontend never supplies a trusted identifier.
- No automated backups are required; the SQLite file location will be documented so it can be copied manually if needed.
- The MVP will not enforce overlapping request validation; two requests from the same employee for overlapping date ranges are permitted in this version.

## 5. Architecture Overview

### 5.1 Architecture Principles

| Principle | Description | Rationale |
|-----------|-------------|-----------|
| Dependency Rule Enforcement | All source dependencies point inward — Domain has no dependencies, Application depends only on Domain, Infrastructure depends on Domain and Application, and Api depends on Application and Infrastructure solely for wiring. | Prevents inner business layers from being contaminated by infrastructure or presentation concerns, enabling the domain and use cases to remain stable when technology choices change. |
| Server-Side Identity Authority | The API always derives the current user and the responsible approver from the authenticated session or token. The frontend never sends a trusted employee or approver identifier for business decisions. | Eliminates the risk of identity spoofing from the frontend, which was identified as a key project risk. Approval responsibility and request ownership are always determined by the authenticated context, not by client-supplied values. |
| Minimal Scope Discipline | The architecture includes only the components required to validate the complete request lifecycle. No optional capabilities, speculative features, or advanced patterns are added beyond what the MVP demands. | Scope creep toward a full HR platform was explicitly identified as a primary risk. Keeping the architecture small and focused ensures the MVP can be delivered and validated without unnecessary complexity. |
| Persistence Ignorance | Domain entities carry no ORM attributes, no EF Core base class dependencies, and no knowledge of how they are stored. All object-relational mapping is configured through Fluent API in the Infrastructure layer. | Keeps the domain model clean and decoupled from the chosen persistence technology, so that storage details can change without affecting business rules or use case logic. |
| Business Rule Enforcement at the API Layer | All business rules — date validations, state transition guards, ownership checks, and manager authorization — are enforced by the API, not only by the UI. The UI may reflect these rules, but cannot be the sole enforcement point. | Ensures that rules cannot be bypassed by manipulating the frontend. Any bypass of authenticated-user identity or approval by a non-manager is an explicit rejection condition for the MVP. |

### 5.2 Architecture Style

> *Auto-populated from Utility Tree sources — do not modify manually.*

**Style**: Reduced Onion Architecture (Domain, Application, Infrastructure, Api, Web)

**Rationale**: Selected via Quality Attribute Weighting analysis. The Utility Tree evaluation identified Security (weight 10) and Reliability (weight 9) as the dominant quality drivers. A reduced Onion Architecture enforces strict inward dependency rules that naturally support both: business rules and authorization logic reside in the Domain and Application layers, which have no dependency on infrastructure or presentation, making them independently testable and immune to contamination from technology choices. The sponsor explicitly requested Clean Architecture principles without unnecessary patterns such as MediatR, CQRS, or generic repositories beyond the basics, which aligns precisely with the reduced form of this pattern. See the Utility Tree document for the full scenario and weight evaluation.

**Key drivers**:
- **Security** — server-side identity authority and role-based authorization enforced at the Application layer, isolated from the frontend
- **Reliability** — state machine and date validation logic concentrated in the Domain layer, enforced by the Application layer regardless of how the API is called
- **Maintainability** — hard layer boundaries make cross-layer violations detectable at code review and prevent infrastructure leakage into business logic

**Trade-offs accepted**:
- Performance is scoped exclusively to single-user local SQLite conditions; no horizontal scaling, caching layer, or load testing is provided
- Compatibility excludes Docker, cloud infrastructure, and all legacy browsers
- Usability addresses functional clarity for an internal MVP tool without formal WCAG certification or screen reader testing
- Compliance and advanced audit logging are deferred to a post-MVP production promotion decision

### 5.3 High-Level Architecture

VacaFlow MVP follows a reduced Onion Architecture with four backend layers and one independent frontend. Dependencies always point inward.

```
┌─────────────────────────────────────────────────────┐
│                   Web (Next.js/React)                │
│  Register · Login · Employee Request List ·          │
│  Request Form · Manager Review List                  │

                        │ HTTP (REST)
┌───────────────────────▼─────────────────────────────┐
│              Api (ASP.NET Core Minimal API)           │

│  ┌───────────────────────────────────────────────┐   │
```

**Key interaction flows:**

- The Next.js frontend communicates with the API exclusively over HTTP. It renders state received from the API and never makes business decisions about identity or authorization.
- The Api layer receives HTTP requests, resolves the authenticated user from the session/token, enforces business rules, and delegates to Application services.
- Application services orchestrate use cases using domain entities and infrastructure ports (repository interfaces). They enforce state transition rules and ownership checks.
- Infrastructure implements persistence (EF Core + SQLite), the repository and Unit of Work, authentication support (password hashing, session or token issuance), and seeds the initial absence types and manager account.
- Domain contains the four core entities, the RequestStatus and Role enumerations, domain exceptions, and business constants. It has no dependencies on any other layer.

### 5.4 Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Authentication mechanism | Application-managed local registration and login with hashed passwords | Microsoft Entra ID and corporate SSO are explicitly deferred for the MVP. Local authentication is the smallest approach that eliminates the previous risk of frontend-supplied user identities and satisfies the requirement that the API always derives the current user from the authenticated session. |
| Database technology | SQLite via Entity Framework Core | The MVP runs locally from source code with no Azure deployment, Docker, or cloud hosting. SQLite keeps the application fully self-contained, requires no server setup, and is sufficient for the small user count and local execution context of the MVP. |
| Backend framework and API style | ASP.NET Core Minimal API | Confirmed by the sponsor and functional analyst as the appropriate backend for this MVP. Minimal API avoids unnecessary scaffolding and aligns with the principle of not adding architectural complexity beyond what the MVP needs. |
| Frontend technology | Next.js and React | Confirmed in scope from the earliest meetings. Provides the web interface for employee and manager workflows with a compact, functional UI limited to the screens required to complete the request lifecycle. |
| Architecture pattern | Reduced Onion Architecture (Domain, Application, Infrastructure, Api) | The sponsor explicitly requested Clean Architecture principles without unnecessary patterns such as MediatR, CQRS, generic repositories beyond the basics, or large frameworks. The reduced Onion structure separates domain concepts, use cases, infrastructure, and API endpoints while keeping the implementation small. |
| Identity resolution for business operations | API derives employee and approver identity exclusively from the authenticated session | The frontend must not send a trusted employee or approver identifier. Allowing the frontend to supply these values was the primary authentication risk identified in the project. All create, submit, cancel, approve, and reject operations use the server-side authenticated user. |
| Manager assignment model | Simple manager reference stored on the Employee record, selected at registration | The MVP does not require a full organizational chart. A registration-time dropdown (or manager checkbox) resolves the assignment. The API checks this assignment when validating approval authorization, without an admin UI or directory integration. |
| Seed data strategy | Seed absence types (Vacation, Personal Leave, Sick Leave) and at least one manager account at database initialization | No catalog maintenance screen is included in the MVP. Seeding ensures a reviewer can run the API and immediately execute the full workflow without manual data setup. |
