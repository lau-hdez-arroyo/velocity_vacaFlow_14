### US-001: User Registration

**As a** new user (employee or manager),
**I want to** register with my full name, email address, password, and role selection,
**So that** I have a personal account in VacaFlow tied to my identity.

**Source Requirements:** FR-AUTH-001, FR-AUTH-002, FR-AUTH-009

**Acceptance Criteria:**

| # | Scenario | Given | When | Then |
|---|----------|-------|------|------|
| AC-001 | Successful employee registration | A user with no existing account opens the registration form and enters a valid name, email, password, and selects the Employee role | They submit the form | The system creates the account with a hashed password, establishes a session, and the user can immediately log in |
| AC-002 | Successful manager registration | A user with no existing account enters valid details and selects the Manager role | They submit the form | The system creates the account with the Manager role and a hashed password; the account is immediately usable for login |
| AC-003 | Duplicate email rejected | A user attempts to register with an email address already present in the database | They submit the registration form | The system rejects the registration with a clear, human-readable error; no account is created |
| AC-004 | Password stored as hash — never plain text | Any user completes a successful registration | The system persists the account | Only the cryptographic hash of the password is written to the database; no plain-text value appears in the database, logs, or any API response |
| AC-005 | Missing required field rejected | A user submits the registration form with one or more required fields empty | The system processes the request | A clear, human-readable validation error is returned identifying the missing field(s); no account is created |
| AC-006 | Database auto-migrated at startup | The API is started against a fresh SQLite file | Startup completes | The schema is fully created and seeded data is present without any manual migration or seed command; the registration endpoint is ready for use |

**Business Rules:** BR-USER-002 (API derives identity from session — frontend never sends trusted ID), BR-DATA-001 (absence type catalog is seeded and not user-maintainable)
**Dependencies:** US-SH-001 (seeded absence types), US-SH-002 (seeded manager account and employee assignment)
**Sprint Assignment:** Sprint 1

### US-002: User Login and Logout

**As a** registered user,
**I want to** log in with my email and password and log out when I am done,
**So that** my session is authenticated and my actions are attributed to me by the API.

**Source Requirements:** FR-AUTH-003, FR-AUTH-004, FR-AUTH-005, FR-AUTH-006

**Acceptance Criteria:**

| # | Scenario | Given | When | Then |
|---|----------|-------|------|------|
| AC-001 | Successful login establishes session | A registered user provides their correct email and password | They submit the login form | The system creates an authenticated session; the current-user endpoint returns the user's name, email, and role |
| AC-002 | Invalid credentials rejected without field disclosure | A user provides an incorrect email or incorrect password | They submit the login form | The system rejects the attempt with a generic authentication error that does not indicate which field was wrong; no session is created |
| AC-003 | Logout invalidates session | An authenticated user triggers logout | The system processes the logout request | The session is terminated; any subsequent request to an authenticated endpoint using the old session credential returns an unauthorized response |
| AC-004 | Current-user endpoint returns session identity | An authenticated session exists | The frontend calls the current-user endpoint | The system returns the authenticated user's name, email, and role derived exclusively from the server-side session |
| AC-005 | Unauthenticated current-user returns 401 | No valid session exists | The current-user endpoint is called | The system returns an unauthenticated response; no user data is disclosed |
| AC-006 | Business endpoints require authentication | No valid session exists | Any business operation endpoint is called | The system returns an unauthorized response and performs no business action |

**Business Rules:** BR-USER-002 (session is the sole identity source)
**Dependencies:** US-001
**Sprint Assignment:** Sprint 1

### US-003: Create Draft Request

**As a** logged-in employee,
**I want to** create a new absence request by selecting an absence type, a start date, an end date, and providing a reason,
**So that** a Draft request is recorded under my identity.

**Source Requirements:** FR-EMP-001, FR-EMP-007, FR-EMP-008

**Acceptance Criteria:**

| # | Scenario | Given | When | Then |
|---|----------|-------|------|------|
| AC-001 | Successful draft creation | An authenticated employee selects a valid absence type (Vacation, Personal Leave, or Sick Leave), a future start date, a valid end date on or after the start date, and provides a reason | They submit the creation form | A new request is persisted in Draft state owned by the authenticated employee; the request appears in the employee's request list |
| AC-002 | Absence type selected from seeded catalog | An authenticated employee opens the request creation form | The form loads | Only the three seeded absence types (Vacation, Personal Leave, Sick Leave) are available for selection; no free-text entry is accepted |
| AC-003 | Start date in the past rejected | An authenticated employee enters a start date that is prior to today's date | They submit the creation form | The API returns a clear, human-readable validation error; no request is created |
| AC-004 | End date before start date rejected | An authenticated employee enters an end date that is earlier than the start date | They submit the creation form | The API returns a clear, human-readable validation error; no request is created |
| AC-005 | Owner derived from session — never from frontend | An authenticated employee creates a request | The system persists the request | The request owner is read exclusively from the authenticated session; any owner identifier supplied in the request body is ignored |
| AC-006 | Missing required field rejected | An authenticated employee omits a required field | They submit the creation form | The API returns a clear, human-readable validation error identifying the missing field; no request is created |

**Business Rules:** BR-REQ-001 (end date ≥ start date), BR-REQ-002 (start date not in past), BR-USER-002 (session identity), BR-DATA-001 (absence type catalog is seeded and not user-maintainable)
**Dependencies:** US-002, US-SH-001 (seeded absence types must exist)
**Sprint Assignment:** Sprint 1

### US-005: Submit Request

**As a** logged-in employee,
**I want to** submit my Draft request for manager review,
**So that** it enters the Submitted state and becomes visible to my assigned manager.

**Source Requirements:** FR-EMP-003, FR-EMP-005

**Acceptance Criteria:**

| # | Scenario | Given | When | Then |
|---|----------|-------|------|------|
| AC-001 | Successful submission transitions request to Submitted | An authenticated employee owns a request in Draft state | They submit the request | The request transitions to Submitted state; it is no longer editable by the employee |
| AC-002 | Submitted request appears in manager review list | An authenticated employee has just submitted a Draft request | Submission completes | The request appears in the review list of the manager assigned to this employee; no other manager sees it |
| AC-003 | Only owner can submit | An authenticated employee attempts to submit a Draft request they do not own | They attempt the submission | The API returns an authorization error; no transition occurs |
| AC-004 | Non-Draft request cannot be submitted | An authenticated employee attempts to submit a request that is not in Draft state | They attempt the submission | The API returns a clear error; no transition occurs |

**Business Rules:** BR-REQ-004 (only owner can submit), BR-STATE-001 (Submitted is read-only for employee; Approved, Rejected, Cancelled are final — no further transitions permitted)
**Dependencies:** US-004, US-SH-002 (manager assignment must exist on the Employee record so the request routes correctly)
**Sprint Assignment:** Sprint 1

### US-007: Manager Reviews and Approves or Rejects a Submitted Request

**As a** logged-in manager,
**I want to** view Submitted requests assigned to my employees and approve or reject each one with an optional comment,
**So that** a decision is recorded with me as the responsible approver and the employee can see the final result.

**Source Requirements:** FR-APPR-001, FR-APPR-002, FR-APPR-003, FR-APPR-004, FR-APPR-005, FR-APPR-006, FR-APPR-007, FR-APPR-008, FR-APPR-009, FR-MGR-REQ-002

**Acceptance Criteria:**

| # | Scenario | Given | When | Then |
|---|----------|-------|------|------|
| AC-001 | Manager sees only assigned employees' Submitted requests | An authenticated manager navigates to the review list | The list loads | Only Submitted requests belonging to employees whose stored `Employee.ManagerId` references this manager are shown; requests from employees assigned to other managers are not visible |
| AC-002 | Successful approval creates Approval record and transitions state | An authenticated manager selects a Submitted request assigned to them and chooses to approve it with an optional comment | They confirm the action | The request transitions to Approved state; exactly one Approval record is created containing the request reference, the authenticated manager's identity (from session), the Approved decision, the optional comment, and the decision date |
| AC-003 | Successful rejection creates Approval record and transitions state | An authenticated manager selects a Submitted request assigned to them and chooses to reject it with an optional comment | They confirm the action | The request transitions to Rejected state; exactly one Approval record is created with the same required fields as AC-002 |
| AC-004 | Approver identity derived from session — never from frontend | A manager approves or rejects a request | The system identifies the responsible approver | The approver's identity is read exclusively from the authenticated session; any approver identifier in the request body is ignored |
| AC-005 | Non-Submitted request cannot be approved or rejected | A manager attempts to approve or reject a request not in Submitted state | They attempt the action | The API returns a clear error; no Approval record is created and no state change occurs |
| AC-006 | Manager cannot decide on unassigned employee's request | A manager attempts to approve or reject a request belonging to an employee not assigned to them | They attempt the action | The API returns an authorization error; no Approval record is created |
| AC-007 | Manager cannot approve or reject their own request | A manager who also has a Submitted request of their own attempts to approve or reject it | They attempt the action | The API returns a clear error; no Approval record is created; the request remains in Submitted state |
| AC-008 | Duplicate Approval record prevented | A manager attempts to approve a request that already has an Approval record | They attempt the action | The API returns an error; no duplicate Approval record is created |

**Business Rules:** BR-MGR-001 (only Submitted requests can be approved or rejected), BR-MGR-003 (only the assigned manager can approve or reject), BR-MGR-004 (manager cannot approve or reject their own request), BR-APPR-001 (exactly one Approval record per decision), BR-APPR-002 (approver identity derived from authenticated session — never from frontend)
**Dependencies:** US-005, US-SH-002 (manager assignment must exist), US-002 (manager must be logged in)
**Sprint Assignment:** Sprint 1

## 4. Definition of Done (Global)

*All stories must meet ALL of the following criteria before being marked "Done":*

- [ ] All acceptance criteria defined for the story pass, verified manually end-to-end using real registered accounts in a locally running instance of the application.
- [ ] All business rules relevant to the story are enforced by the API, not only by the UI — including ownership checks, state transition guards, date validation, and manager authorization (NFR-SEC-003).
- [ ] The API derives the current user identity and the responsible approver exclusively from a server-side session or token mechanism (cookie or JWT — implementation's choice per SAD §5.4 / Assumption A-01). No trusted employee ID or approver ID is accepted from the frontend for any business decision.
- [ ] Passwords are stored as hashes. No plain-text credential appears in the database, logs, or any API response.
- [ ] Invalid operations return clear, human-readable, structured error responses — not generic 500 errors. Field-level messages are returned for validation failures.
- [ ] The story does not introduce any functionality from the explicitly deferred Won't v1 backlog (no notifications, no balance calculations, no admin screens, no external integrations, no overlapping validation, no Employee-to-Manager assignment UI or self-service assignment — OS-019).
- [ ] The code compiles and the application starts cleanly from source with a freshly initialized SQLite database, including seeded absence types (Vacation, Personal Leave, Sick Leave) and at least one seeded manager account with an assigned employee — with no manual migration or seed command.
- [ ] The implemented behavior does not bypass or contradict any previously accepted story's acceptance criteria or any of the signed business rules (FRS-001 §5).
- [ ] The Product Owner (Laura Hernandez) or designated reviewer has reviewed and accepted the story during the MVP validation window.
