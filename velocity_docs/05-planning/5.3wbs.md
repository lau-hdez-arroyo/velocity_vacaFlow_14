## 1. Introduction

### 1.1 Purpose

This Work Breakdown Structure (WBS) decomposes the VacaFlow_14 project scope into manageable deliverables and work packages. It serves as the authoritative scope baseline for Sprint 1 execution, governance tracking, and effort estimation. All work packages map directly to the 10 user stories in BLG-001, the milestones in EA-001, and the components in SAD-DD-001.

### 1.2 Project Summary

VacaFlow_14 is a locally executed MVP that enables employees to create, submit, edit, and cancel absence requests and enables managers to approve or reject submitted requests with a recorded decision. The system runs on a two-command startup (`dotnet run` + `npm run dev`) with a SQLite-backed ASP.NET Core API and a Next.js React frontend. The entire delivery — documentation, implementation, hardening, and acceptance sign-off — is compressed into a 5-day window (2026-07-18 to 2026-07-22), with implementation and acceptance on a single day (2026-07-22).

### 1.3 Scope Boundaries

**In Scope (BLG-001 Must Have + Should Have):**
- User registration, login, and logout (US-001, US-002)
- Seeded absence type catalog and manager account (US-SH-001, US-SH-002)
- Draft request creation, editing, submission, and cancellation (US-003, US-004, US-005, US-006)
- Manager approval and rejection workflow with decision record (US-007)
- Employee view of final decision and comment (US-008)

**Explicitly Out of Scope (BLG-001 Won't v1 — 15 items):**
All 15 items from BLG-001 §3 are excluded: corporate SSO, Azure deployment, Docker/CI-CD, email/Teams notifications, password reset, account administration UI, vacation balance calculation, holiday calendar, overlapping request validation, attachments, reports/exports, HR views and multi-level approvals, approval delegation, external integrations, and advanced audit logs. No Won't v1 item may enter scope without written approval from James Parker (Sponsor) and Laura Hernandez (Product Owner), with an equivalent-size offset from the Won't v1 list.

### 1.4 WBS Approach

- **Deliverable-oriented decomposition:** work packages represent outputs, not activities in isolation.
- **100% rule:** all project work — documentation, implementation, testing, deployment, and closure — is captured.
- **Mutually exclusive packages:** no work package overlaps with another.
- **8/80 rule:** work packages are sized between 8 and 80 person-hours where estimable; the compressed single-day sprint window means several packages are tightly co-located in time.
- **Sprint-sequence alignment:** implementation work packages follow the BLG-001 sprint sequence (seeded data first, then auth, then request workflow, then approval).
