### Recommendation

VacaFlow_14 demonstrates a sound security baseline for an MVP: passwords are hashed with BCrypt, user identity is derived server-side from the session, and business-rule enforcement is structurally embedded in the vertical slice architecture. The three High findings — HTTPS enforcement, anti-CSRF token delivery, and SQLite database file exposure controls — must be addressed before any shared or semi-public deployment. The five Medium findings represent improvements to defense-in-depth posture and should be scheduled in the first post-MVP sprint. No formal compliance standard is in scope for this MVP; basic security hygiene requirements as identified in project documents are assessed below.

## 3. High Severity Findings

### SEC-001: HTTPS Not Enforced in Local Development Configuration

| Attribute | Value |
|-----------|-------|
| **ID** | SEC-001 |
| **Severity** | High |
| **Category** | Transport Security |
| **Status** | Open |
| **CVSS Score** | 7.4 (AV:N/AC:H/PR:N/UI:N/S:U/C:H/I:H/A:N) |
| **Affected Component** | ASP.NET Core API, Next.js Frontend |

**Description:**
The local development configuration uses plain HTTP by default. Although this is standard for local-only development, the absence of enforced HTTPS means that if the application is accessed over a shared network (e.g., during a demo, acceptance walkthrough, or LAN-accessible deployment), credentials, session cookies, and all request data are transmitted in cleartext. The ASP.NET Core middleware pipeline does not include `UseHttpsRedirection()` as a mandatory, validated step, and the Next.js proxy configuration does not enforce TLS upstream.

**Impact:**
An attacker on the same network segment could perform a passive eavesdropping or active man-in-the-middle attack, intercepting session cookies, password submissions, and approval workflow data. HTTP-only and SameSite=Lax cookie attributes provide no protection against network-layer interception.

**Recommendation:**
Enable HTTPS for all environments where the application is accessible beyond localhost. At minimum, configure `UseHttpsRedirection()` and `UseHsts()` in the ASP.NET Core pipeline and document HTTPS setup in the README. For the acceptance walkthrough, verify that the environment is localhost-only. Before any shared deployment, obtain a valid TLS certificate and enforce HTTPS at the reverse-proxy or Kestrel layer.

**References:**
- OWASP: Sensitive Data Exposure (A02:2021)
- CWE-319: Cleartext Transmission of Sensitive Information

### SEC-002: Anti-Forgery Token Delivery Mechanism Not Verified

| Attribute | Value |
|-----------|-------|
| **ID** | SEC-002 |
| **Severity** | High |
| **Category** | CSRF Protection |
| **Status** | Open |
| **CVSS Score** | 7.1 (AV:N/AC:H/PR:N/UI:R/S:U/C:H/I:H/A:N) |
| **Affected Component** | Anti-Forgery Middleware, Next.js Frontend |

**Description:**
SAD-DD-001 specifies that Anti-Forgery Middleware validates XSRF tokens on all state-changing requests (POST, PUT, PATCH, DELETE). However, the assessment was limited to manual workflow validation and architecture review; the mechanism by which the Next.js frontend obtains, stores, and transmits the XSRF token on every state-changing call was not confirmed through code inspection or dynamic testing. If the anti-forgery token is not correctly wired end-to-end — for example, if the Next.js client does not read the antiforgery cookie and include it as a request header — the CSRF protection is effectively disabled despite the middleware being registered.

**Impact:**
A successful CSRF attack could allow a malicious website to trigger state-changing API calls (approve/reject requests, submit requests, cancel requests) on behalf of an authenticated user without their knowledge, exploiting the cookie-based session.

**Recommendation:**
Perform code-level verification that: (a) the ASP.NET Core API issues the antiforgery token as a readable (non-HTTP-only) cookie on initial page load, (b) the Next.js frontend reads that cookie and attaches its value as a request header (e.g., `X-XSRF-TOKEN`) on every mutating fetch call, and (c) the middleware correctly validates that header. Add an automated integration test (via `WebApplicationFactory`) that submits a state-changing request without the XSRF token and asserts a 400 or 403 response.

**References:**
- OWASP: CSRF (A01:2021 — Broken Access Control, cross-site request category)
- CWE-352: Cross-Site Request Forgery

### SEC-003: SQLite Database File Exposure and Credential Controls

| Attribute | Value |
|-----------|-------|
| **ID** | SEC-003 |
| **Severity** | High |
| **Category** | Data Exposure / File Security |
| **Status** | Open |
| **CVSS Score** | 7.5 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N) |
| **Affected Component** | SQLite Database, SeedDataInitializer, Source Control Configuration |

**Description:**
The SQLite database file stores all application data including employee email addresses, BCrypt password hashes, full names, absence request reasons, approval comments, and dates. ADR-004 specifies that the `.db` file must be excluded from source control via `.gitignore`. The assessment identified the following specific risks: (a) the `.gitignore` rule for the `.db` file has not been independently verified; (b) the SeedDataInitializer seeds at least one Manager account with "controlled, non-real credentials" — if those seed credentials are hardcoded in committed source code, they represent a credential exposure; (c) the database file path and permissions at the operating-system level are not documented, meaning a misconfigured local setup could leave the file world-readable.

**Impact:**
If the database file is committed to source control, all password hashes, personal data, and workflow history are exposed to anyone with repository access. If seed credentials are hardcoded in committed code, any developer can authenticate as a pre-seeded privileged account. Exposed password hashes, even BCrypt, remain subject to offline cracking attempts.

**Recommendation:**
(a) Verify and test the `.gitignore` rule to confirm the `.db` file is excluded, including any common alternative filenames or paths. (b) Move seed credentials out of committed source code into environment variables, a `.env` file (gitignored), or `appsettings.Development.json` (gitignored). (c) Document the recommended file-system permissions for the database file in the README (e.g., owner-read-write only). (d) Add a pre-commit hook or CI check that fails if a `.db` file or known credential pattern is detected in the commit.

**References:**
- CWE-312: Cleartext Storage of Sensitive Information
- CWE-798: Use of Hard-coded Credentials
- OWASP: A02:2021 Cryptographic Failures

## 4. Medium Severity Findings

### SEC-004: Password Policy Not Enforced at Registration

| ID | SEC-004 |
|----|---------|
| **Severity** | Medium |
| **Category** | Authentication |
| **Status** | Open |

**Description:** The `RegisterSlice` hashes passwords with BCrypt (satisfying NFR-SEC-001), but no minimum password complexity or length requirement was confirmed to be enforced by the `IValidator<RegisterCommand>`. A user could register with a single-character password, which BCrypt will hash but which provides negligible brute-force resistance.

**Recommendation:** Add a validation rule in the `RegisterCommand` validator enforcing a minimum password length (recommended: 12 characters) and optionally a character class requirement. Log a validation error with a human-readable message if the constraint is not met.

### SEC-005: Manager-Employee Assignment Gap Silently Hides Requests

| ID | SEC-005 |
|----|---------|
| **Severity** | Medium |
| **Category** | Authorization |
| **Status** | Open |

**Description:** SAD-DD-001 §3.3.5 explicitly notes that "a missing assignment silently hides that employee's requests from all managers." If an `Employee.ManagerId` is null (because the seed or registration flow did not assign a manager), the employee's submitted requests never appear in any manager's queue. No error or warning is surfaced to the employee or any administrator. This is compounded by open item R-01 (Manager-Employee assignment cardinality pending Sponsor confirmation). Additionally, the `ApproveRequestSlice` enforces the manager-assignment check (returns 403 if not assigned), meaning if the assignment is missing, approval is blocked with no actionable feedback to the employee.

**Recommendation:** Add a server-side check at request submission time that validates the submitting employee has a non-null `ManagerId`. If no manager is assigned, return a structured 422 error with a human-readable message (e.g., "No manager is assigned to your account. Contact an administrator before submitting a request."). Resolve open item R-01 before the acceptance Gate G4 sign-off.

### SEC-006: Stack Traces and Internal Details in 500 Responses

| ID | SEC-006 |
|----|---------|
| **Severity** | Medium |
| **Category** | Error Handling / Information Disclosure |
| **Status** | Open |

**Description:** The `GlobalExceptionHandler` is specified to map all unhandled exceptions to RFC 7807 Problem Details and to suppress stack traces in 500 responses. However, this behavior was validated only through architecture review, not through dynamic testing of unhandled exception scenarios. ASP.NET Core's default developer exception page, if enabled via `app.UseDeveloperExceptionPage()` in the Development environment, would override the `GlobalExceptionHandler` and expose full stack traces, internal file paths, and source code snippets to the client.

**Recommendation:** Explicitly verify that `UseDeveloperExceptionPage()` is gated behind an `if (app.Environment.IsDevelopment())` block and is not reachable in the acceptance or production environment. Add an integration test that deliberately triggers an unhandled exception and asserts the response body does not contain stack trace keywords (e.g., `at System.`, `StackTrace`). Document this check in the pre-acceptance checklist.

### SEC-007: Absence of Security Response Headers

| ID | SEC-007 |
|----|---------|
| **Severity** | Medium |
| **Category** | Security Headers |
| **Status** | Open |

**Description:** The ASP.NET Core API and Next.js frontend are not confirmed to emit security-relevant HTTP response headers: `Content-Security-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy`. For a local MVP these represent defense-in-depth controls, but their absence creates risk if the application is demo'd in a shared or semi-public network context.

**Recommendation:** Add a middleware call to `UseSecurityHeaders()` (or configure headers directly in `app.Use`) that emits at minimum: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, and `Referrer-Policy: strict-origin-when-cross-origin`. For the Next.js frontend, configure `next.config.js` `headers()` to emit equivalent headers on all routes. Implementing a strict Content-Security-Policy is recommended as a post-MVP improvement.

---

### SEC-008: Input Length Limits Not Confirmed on All Fields

| ID | SEC-008 |
|----|---------|
| **Severity** | Medium |
| **Category** | Input Validation |
| **Status** | Open |

**Description:** The `ValidationPipelineInvoker` guarantees that all registered `IValidator<T>` instances run before any handler. However, the assessment could not confirm that validators for all slice commands enforce maximum field-length constraints on free-text fields: `Reason` (BR-FIELD-001), approval `Comment`, `FullName`, and `Email`. Without upper bounds, an attacker could submit excessively long strings that consume storage or trigger application errors.

**Recommendation:** Review all `IValidator<T>` implementations to confirm that every free-text field has an explicit maximum length rule (recommended: Reason and Comment ≤ 1000 characters; FullName ≤ 200 characters; Email ≤ 254 characters per RFC 5321). Ensure EF Core column mappings enforce matching `HasMaxLength()` constraints at the database level.

## 5. Low Severity Findings

### SEC-009: No Account Lockout After Repeated Failed Login Attempts

| ID | SEC-009 |
|----|---------|
| **Severity** | Low |
| **Category** | Authentication |
| **Status** | Open |

**Description:** The `LoginSlice` validates credentials and issues a cookie session, but no rate-limiting or account lockout mechanism was identified for repeated failed login attempts. For a local MVP with a controlled user base, this risk is low. If the application is made accessible over a network, brute-force attacks become feasible.

**Recommendation:** Add rate-limiting on `POST /auth/login` using ASP.NET Core's built-in `RateLimiter` middleware (available in .NET 7+). For post-MVP deployments, consider temporary lockout after a configurable number of failed attempts (e.g., 10 failures within 15 minutes).

---

### SEC-010: Cookie Expiry and Session Lifetime Not Documented

| ID | SEC-010 |
|----|---------|
| **Severity** | Low |
| **Category** | Authentication |
| **Status** | Open |

**Description:** The ASP.NET Core cookie session is correctly invalidated on explicit logout (NFR-SEC-005). However, the session cookie expiry and idle timeout configuration were not confirmed in the architecture review. An excessively long session lifetime (e.g., 30-day persistent cookie) could allow a stolen cookie to remain valid long after the user intended to end their session.

**Recommendation:** Document and configure explicit session lifetime parameters in the `CookieAuthenticationOptions`: set `ExpireTimeSpan` to a reasonable bound (e.g., 8 hours for a workday context), `SlidingExpiration = true` for active users, and `IsPersistent = false` to ensure the cookie is a session cookie that does not persist beyond the browser session.

---

### SEC-011: Security Events Not Written to a Persistent Audit Log

| ID | SEC-011 |
|----|---------|
| **Severity** | Low |
| **Category** | Logging and Monitoring |
| **Status** | Open |

**Description:** The `GlobalExceptionHandler` logs unhandled exceptions via `ILogger`, and the architecture describes structured RFC 7807 responses. However, no dedicated security event log was identified for authentication events (successful login, failed login, logout), authorization failures (403 responses from ownership or assignment checks), and state-machine violations (422 responses). For the MVP, console logging may be sufficient. For any shared deployment, the absence of a persistent security audit trail limits incident investigation capability.

**Recommendation:** Add structured security event logging to the following slice handlers: `LoginSlice` (log success/failure with timestamp and email, never password), `LogoutSlice` (log success), `ApproveRequestSlice` / `RejectRequestSlice` (log authorization failures and decisions), and `GlobalExceptionHandler` (log 403 and 422 events). Direct these log entries to a persistent sink (file, structured log) rather than console-only output.

## 6. Informational Finding

### SEC-012: Open Item R-01 Has Downstream Security Implications

| ID | SEC-012 |
|----|---------|
| **Severity** | Informational |
| **Category** | Authorization |
| **Status** | Open |

**Description:** SAD-DD-001 §1.1 records that the Manager-to-Employee assignment cardinality (`Employee.ManagerId`, BR-013, BR-DATA-002, FR-AUTH-010) is pending formal confirmation by James Parker (Sponsor) and constitutes the one outstanding blocking technical open item under Gate G4. Until this is resolved, the authorization path for manager-scoped request visibility (`GetRequestsSlice`) and manager-assignment checks in `ApproveRequestSlice` / `RejectRequestSlice` relies on an unconfirmed design assumption. If the cardinality model changes (e.g., an employee can have multiple managers, or no manager), the authorization logic in multiple slices requires revision.

**Recommendation:** Escalate open item R-01 to the Sponsor (James Parker) for formal sign-off before Gate G4. Document the confirmed cardinality model in BR-001 v1.3. If the model changes, update `ApproveRequestSlice`, `RejectRequestSlice`, `GetRequestsSlice`, and `SeedDataInitializer` accordingly and re-assess SEC-005.
