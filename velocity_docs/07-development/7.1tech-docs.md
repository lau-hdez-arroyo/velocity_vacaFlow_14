## 1. Introduction

### 1.1 Purpose

This document provides comprehensive technical guidance for developers working on VacaFlow_14, a local-execution web application for managing employee absence and vacation requests. It covers the system architecture, development environment setup, project structure, coding patterns, testing approach, and troubleshooting guidance.

VacaFlow_14 implements a defined request lifecycle — Draft → Submitted → Approved / Rejected / Cancelled — enforced by a Domain-layer state machine. The MVP is designed to run entirely from source with two terminal commands and no external infrastructure beyond the developer's workstation.

### 1.2 Audience

- Backend and frontend developers joining the project
- Reviewers running the 11-step acceptance walkthrough
- Technical support staff maintaining the local deployment
- New team members onboarding to the codebase

### 1.3 Scope

This document covers the VacaFlow_14 MVP (Phase 1 — Acceptance Build). The MVP is a self-contained local application with no cloud deployment, no Docker, and no external service integrations. Post-MVP evolution paths (mobile clients, JWT migration, cloud deployment) are noted where relevant but are out of scope for this document.

### 1.4 Prerequisites

Developers must have familiarity with:

- **Backend:** C# and .NET 8 (ASP.NET Core), Entity Framework Core, and the Onion Architecture pattern
- **Frontend:** TypeScript, React 18, and Next.js 14
- **General:** RESTful API design, HTTP cookie-based authentication, and relational database concepts

Access requirements:

- Source code repository access (read/write)
- A workstation running Windows, macOS, or Linux with internet access for package restoration on first build

## 2. Architecture Overview

### 2.1 High-Level Architecture

VacaFlow_14 is composed of three runtime containers:

```
┌────────────────────────────────────────────────────────────────────┐
│  Browser (Employee or Manager)                                     │
└───────────────────────┬────────────────────────────────────────────┘
                        │ HTTPS
┌───────────────────────▼────────────────────────────────────────────┐
│  Next.js Frontend (Next.js 14 / React 18 / TypeScript)             │
│  Port 3000 (dev server)                                            │
│  Proxies /api/* → ASP.NET Core API                                 │
└───────────────────────┬────────────────────────────────────────────┘
                        │ HTTP (localhost proxy) / JSON
┌───────────────────────▼────────────────────────────────────────────┐
│  ASP.NET Core API (.NET 8 / C# / EF Core 8)                        │
│  Port 5000 (dev) / Port 5001 (HTTPS dev)                           │
│  Vertical Slice Onion Architecture                                  │
│  Cookie Session Auth + Anti-Forgery + Validation Pipeline          │
│  Domain-layer RequestStateMachine                                   │
└───────────────────────┬────────────────────────────────────────────┘
                        │ EF Core / SQLite ADO.NET
┌───────────────────────▼────────────────────────────────────────────┐
│  SQLite Database (file-based, auto-created at startup)             │
│  vacaflow.db (local file, not in source control)                   │
└────────────────────────────────────────────────────────────────────┘
```

### 2.2 Architectural Style

VacaFlow_14 uses a **Pragmatic Vertical Slice Onion** architecture (Alternative C, selected per AED-001 / GOV-001 Decision Log D002, documented in SAD-DD-001 ADR-001). The key organizing principle is one file per operation — each business operation (e.g., `CreateRequestSlice.cs`, `ApproveRequestSlice.cs`) is a self-contained vertical slice that contains the endpoint definition, the slice handler, and the inline validator for that operation.

The Onion layering governs dependency direction:

```
┌──────────────────────────────────────────────────┐
│  Infrastructure (EF Core, SQLite, BCrypt, Seed)  │
│  ┌────────────────────────────────────────────┐  │
│  │  Application (Slices, ValidationPipeline)  │  │
│  │  ┌──────────────────────────────────────┐  │  │
│  │  │  Domain (RequestStateMachine, Ports) │  │  │
│  │  └──────────────────────────────────────┘  │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

Domain has zero dependencies on Application, Infrastructure, or any external library. Infrastructure depends on Domain and Application. The Api layer (ASP.NET Core Minimal API) and Web layer (Next.js) are the two outermost entry points.

## 3. Technical Constraints

> **Source:** Extracted from SAD-DD-001 (Technical Constraints throughout §1–§4) and the Architecture Overview (SAD-001 v1.2, architectural principles and design decisions). These constraints are mandatory and must guide all implementation decisions without exception.

### 3.1 Technical Constraints (from Software Architecture Document — SAD-DD-001)

| Constraint | Category | Impact on Implementation |
|------------|----------|--------------------------|
| No Docker, no external database server, no cloud services | Compatibility (NFR-COMPAT-001) | SQLite is the only permitted persistence store; the API and frontend must run directly from source with `dotnet run` and `npm run dev` |
| Cold start (migration + seeding + API ready) must complete in < 10 seconds | Availability (SC-07, SC-14) | EF Core migrations and `SeedDataInitializer` must execute at startup without external dependency; no boot-time validation gate that could abort startup |
| Logout must invalidate the session — no old credential re-use after `POST /auth/logout` | Security (NFR-SEC-005, Critical) | ASP.NET Core Cookie Session with `HttpContext.SignOutAsync()` is mandatory; JWT is prohibited for the MVP; the server-side session is the single source of auth truth |
| BCrypt password hashing is required for all stored credentials | Security (NFR-SEC-001, SC-01) | `IPasswordHasher` port backed by `BCryptHasher` adapter must be invoked exclusively by `RegisterSlice` and `LoginSlice`; no plain-text credential may be written to the database |
| Identity must be read from `HttpContext.User` exclusively | Security (NFR-SEC-002, SC-02) | Slice handlers must never accept a userId from the request body or query parameters; all identity-dependent authorization checks derive from the encrypted cookie claim |
| A new developer must be able to start the application from the README in ≤ 15 minutes | Maintainability (SC-10, NFR-MAINT-004) | Repository README must document the exact commands to restore, migrate, and run both applications; no secret-generation step is permitted for the MVP |
| MediatR is prohibited | Maintainability (NFR-MAINT-002) | The validation pipeline must be hand-rolled using the `ValidationPipelineInvoker` + `IValidator<T>` port pattern; no MediatR package may be referenced |
| Domain layer must carry zero dependencies on EF Core, ASP.NET Core, or any external library | Maintainability (SC-09, NFR-MAINT-003) | The `RequestStateMachine` static class and all Domain entities must have no using directives or package references pointing to Application, Infrastructure, or any external NuGet package |
| Approval records must be created atomically with the corresponding state transition | Reliability (NFR-REL-003) | `ApproveRequestSlice` and `RejectRequestSlice` must persist the status change and the `Approval` record in a single `SaveChangesAsync()` call — never in two separate commits |
| Data must persist across API restarts | Reliability (SC-06, NFR-REL-004) | SQLite file-based storage and EF Core idempotent migrations satisfy this; the DB file must not be in a temp directory; the README must document the path |
| Manager authorization checks (role, assignment, self-approval prohibition) must execute unconditionally before domain logic | Security (NFR-SEC-003, SC-03) | These checks must be inside `ApproveRequestSlice` and `RejectRequestSlice` handlers, not delegated to middleware that could be accidentally bypassed |
| The SQLite database file must be excluded from source control | Security (NFR-SEC-004) | `.gitignore` must include the `.db` file pattern |

### 3.2 Architectural Constraints (from Architecture Overview — SAD-001 v1.2)

| Constraint | Origin | Implication |
|------------|--------|-------------|
| Vertical slice structure: one file per operation | Alternative C selection (ADR-001, SAD-DD-001) | Every new business operation must be a new slice file; use-case service classes are prohibited; cross-slice shared logic must be extracted to `Application/` or `Domain/` shared components, never duplicated |
| `RequestStateMachine` must reside in the Domain layer as a static class | ADR-002, SAD-001 v1.2 baseline | The static class must have no DI registration; slice handlers call it directly via a static method invocation; moving it to Application or Infrastructure breaks the architectural baseline and fails SC-04 |
| ASP.NET Core Cookie Session Authentication with HTTP-only, SameSite=Lax cookies | ADR-003, SAD-DD-001 §4.2 | JWT Bearer is prohibited for the MVP; the cookie must be configured as `HttpOnly = true`, `SameSite = SameSiteMode.Lax`; the `next.config.js` proxy must forward `/api` to the ASP.NET Core backend so cookies share the same origin |
| Anti-forgery middleware must be configured globally on all state-changing endpoints | ADR-003, AED-001 §7 | `POST`, `PUT`, `PATCH`, and `DELETE` endpoints must validate the XSRF token; the Next.js frontend must read and resend the antiforgery token on every state-changing request |
| `ValidationPipelineInvoker` must run before every handler — structurally guaranteed, not by convention | ADR-005, SAD-DD-001 §4.5 | Every slice's `IValidator<T>` must be registered in DI at startup; the invoker must be wired into the endpoint pipeline before the handler delegate; a new slice without a registered validator fails the architectural contract |
| EF Core Code-First with automatic migration at startup | ADR-004, SAD-DD-001 §4.4 | `MigrateAsync()` must be called in the application startup sequence before the API begins accepting requests; `SeedDataInitializer` must run after migration, idempotently |
| Persistence Ignorance: Domain entities must carry no ORM attributes | SAD-001 v1.2 §5.1 | Fluent API configuration in `VacaFlowDbContext.OnModelCreating()` is the sole source of EF Core mapping; no `[Column]`, `[Table]`, `[Key]`, or other EF attributes on Domain entity classes |
| Exactly one `Approval` record per approval decision, enforced at the database level | NFR-REL-003, SAD-DD-001 §3.1 | The EF Core model must configure a unique constraint on `Approval.RequestId`; the `ApproveRequestSlice` and `RejectRequestSlice` must check for an existing `Approval` record before inserting |

### 3.3 Implementation Guidelines Derived from Constraints

Based on the constraints above, all developers MUST adhere to the following rules during implementation:

- **One slice file per operation.** Do not create service classes in the Application layer for VacaFlow_14 operations. If a concern is shared across multiple slices, extract it to a shared helper in `Application/Shared/` or `Domain/`.
- **Call `RequestStateMachine.Transition()` before `SaveChangesAsync()`.** Every state-changing slice must invoke the Domain state machine first; the `DomainException` it raises on an invalid transition will be caught and mapped to HTTP 422 by `GlobalExceptionHandler`.
- **Read identity from `HttpContext.User` only.** Never bind a `userId`, `managerId`, or `employeeId` from the request body or route parameters for authorization decisions.
- **Register an `IValidator<T>` for every new command type.** The `ValidationPipelineInvoker` is only as effective as the validators registered against it. A slice without a registered validator silently bypasses validation.
- **Never reference EF Core types in the Domain layer.** Do not add `using Microsoft.EntityFrameworkCore;` to any file under `Domain/`. EF Core belongs exclusively in `Infrastructure/`.
- **Use a single `SaveChangesAsync()` per handler.** Each slice handler must commit exactly once at the end of the happy path. Never call `SaveChangesAsync()` multiple times within the same request.
- **Hash passwords with `IPasswordHasher` only.** Do not call BCrypt directly from a slice. Inject `IPasswordHasher` and delegate; this preserves testability and the Domain port contract.
- **Exclude `vacaflow.db` from version control.** Confirm the `.gitignore` contains `*.db` or the specific filename before committing for the first time.

> **Note:** Any deviation from the constraints in Sections 3.1 and 3.2 requires written approval from the Solution Architect and must be documented as a new Architecture Decision Record (ADR) in SAD-DD-001.

## 6. Development Guide

### 6.1 Coding Patterns

#### 6.1.1 Vertical Slice Pattern

Each business operation is a single self-contained C# file containing the endpoint definition, the command/request record, the inline validator, and the handler. The following example shows `CreateRequestSlice.cs`.

```csharp
// Api/Slices/Requests/CreateRequestSlice.cs

using Microsoft.AspNetCore.Mvc;
using VacaFlow.Application.Validation;
using VacaFlow.Domain.Enums;
using VacaFlow.Infrastructure.Persistence;
// Domain entities imported — no EF Core using here

namespace VacaFlow.Api.Slices.Requests;

public static class CreateRequestSlice
{
    // Command record — bound from the HTTP request body
    public record CreateRequestCommand(
        Guid AbsenceTypeId,
        DateOnly StartDate,
        DateOnly EndDate,
        string Reason
    );

    // Inline validator — registered in DI at startup
    public class CreateRequestValidator : IValidator<CreateRequestCommand>
    {
        public ValidationResult Validate(CreateRequestCommand command)
        {
            var errors = new List<ValidationError>();

            if (command.AbsenceTypeId == Guid.Empty)
                errors.Add(new("absenceTypeId", "Absence type is required."));

            if (command.StartDate < DateOnly.FromDateTime(DateTime.UtcNow.Date))
                errors.Add(new("startDate", "Start date cannot be in the past."));

            if (command.EndDate < command.StartDate)
                errors.Add(new("endDate", "End date must be on or after the start date."));

            if (string.IsNullOrWhiteSpace(command.Reason))
                errors.Add(new("reason", "Reason is required."));

            return errors.Count == 0
                ? ValidationResult.Success()
                : ValidationResult.Failure(errors);
        }
    }

    // Endpoint registration — called from SliceRegistrationExtensions
    public static void Register(IEndpointRouteBuilder app)
    {
        app.MapPost("/api/requests", HandleAsync)
           .RequireAuthorization();
    }

    // Handler — only reached after ValidationPipelineInvoker passes
    private static async Task<IResult> HandleAsync(
        [FromBody] CreateRequestCommand command,
        HttpContext httpContext,
        VacaFlowDbContext db,
        CancellationToken ct)
    {
        // Identity is always read from HttpContext.User — never from the request body
        var ownerIdClaim = httpContext.User.FindFirst("sub")?.Value
            ?? throw new UnauthorizedAccessException("Authenticated employee identity not found.");
        var ownerId = Guid.Parse(ownerIdClaim);

        // Verify the AbsenceType exists (catalog reference check)
        var absenceType = await db.AbsenceTypes.FindAsync([command.AbsenceTypeId], ct)
            ?? throw new NotFoundException("The selected absence type does not exist.");

        var request = new AbsenceRequest
        {
            Id = Guid.NewGuid(),
            OwnerEmployeeId = ownerId,
            AbsenceTypeId = command.AbsenceTypeId,
            StartDate = command.StartDate,
            EndDate = command.EndDate,
            Reason = command.Reason,
            Status = RequestStatus.Draft
        };

        db.Requests.Add(request);
        await db.SaveChangesAsync(ct); // Single commit — satisfies NFR-REL-003

        return Results.Created($"/api/requests/{request.Id}", request.ToDto());
    }
}
```

**Rules for every new slice:**
1. One file per operation under `Api/Slices/{Category}/`
2. Define a `record` command/query type at the top of the file
3. Implement `IValidator<TCommand>` inline in the same file
4. Register the validator in DI at startup (see Section 6.2)
5. Call `db.SaveChangesAsync()` exactly once per handler happy path

#### 6.1.2 Domain State Machine Pattern

`RequestStateMachine` is a static class in the Domain layer. Slice handlers call it directly — no injection required.

```csharp
// Domain/StateMachine/RequestStateMachine.cs

using VacaFlow.Domain.Entities;
using VacaFlow.Domain.Enums;
using VacaFlow.Domain.Exceptions;

namespace VacaFlow.Domain.StateMachine;

public static class RequestStateMachine
{
    // Valid transitions: (current status, action) → next status
    private static readonly Dictionary<(RequestStatus, RequestAction), RequestStatus> Transitions = new()
    {
        [(RequestStatus.Draft, RequestAction.Submit)]   = RequestStatus.Submitted,
        [(RequestStatus.Draft, RequestAction.Cancel)]   = RequestStatus.Cancelled,
        [(RequestStatus.Submitted, RequestAction.Approve)] = RequestStatus.Approved,
        [(RequestStatus.Submitted, RequestAction.Reject)]  = RequestStatus.Rejected,
        [(RequestStatus.Submitted, RequestAction.Cancel)]  = RequestStatus.Cancelled,
    };

    /// <summary>
    /// Applies the given action to the request, mutating its Status.
    /// Throws DomainException if the transition is not valid.
    /// </summary>
    public static void Transition(AbsenceRequest request, RequestAction action)
    {
        var key = (request.Status, action);
        if (!Transitions.TryGetValue(key, out var nextStatus))
        {
            throw new DomainException(
                $"Cannot apply '{action}' to a request in '{request.Status}' state.");
        }
        request.Status = nextStatus;
    }

    /// <summary>
    /// Returns the set of valid actions for the given status.
    /// Used by the API to populate UI action hints.
    /// </summary>
    public static IReadOnlyList<RequestAction> GetValidActions(RequestStatus status)
        => Transitions.Keys
            .Where(k => k.Item1 == status)
            .Select(k => k.Item2)
            .ToList()
            .AsReadOnly();
}
```

#### 6.1.3 Approval Slice Authorization Pattern

`ApproveRequestSlice` and `RejectRequestSlice` must enforce all three authorization checks — unconditionally — before invoking the state machine.

```csharp
// Api/Slices/Approvals/ApproveRequestSlice.cs (authorization section)

private static async Task<IResult> HandleAsync(
    Guid id,
    [FromBody] ApproveRequestCommand command,
    HttpContext httpContext,
    VacaFlowDbContext db,
    CancellationToken ct)
{
    // 1. Identity — always from HttpContext.User
    var managerId = Guid.Parse(
        httpContext.User.FindFirst("sub")?.Value
        ?? throw new UnauthorizedAccessException());

    // 2. Load the manager entity and verify the Manager role (BR-MGR-002 / NFR-SEC-003)
    var manager = await db.Employees.FindAsync([managerId], ct)
        ?? throw new UnauthorizedAccessException("Manager account not found.");
    if (manager.Role != Role.Manager)
        throw new AuthorizationException("Only users with the Manager role can approve requests.");

    // 3. Load the request
    var request = await db.Requests
        .Include(r => r.OwnerEmployee)
        .FirstOrDefaultAsync(r => r.Id == id, ct)
        ?? throw new NotFoundException($"Request {id} not found.");

    // 4. Verify manager assignment (FR-APPR-008 / BR-DATA-002)
    if (request.OwnerEmployee.ManagerId != managerId)
        throw new AuthorizationException("You are not the assigned manager for this request.");

    // 5. Self-approval prohibition (FR-APPR-009 / BR-MGR-004)
    if (request.OwnerEmployeeId == managerId)
        throw new AuthorizationException("Managers cannot approve their own requests.");

    // 6. State machine — after all authorization checks
    RequestStateMachine.Transition(request, RequestAction.Approve);

    // 7. Create Approval record
    var approval = new Approval
    {
        Id = Guid.NewGuid(),
        RequestId = request.Id,
        DecidedByEmployeeId = managerId,
        Decision = ApprovalDecision.Approved,
        DecidedAt = DateTime.UtcNow,
        Comment = command.Comment
    };
    db.Approvals.Add(approval);

    // 8. Single atomic commit — state transition + Approval record (NFR-REL-003)
    await db.SaveChangesAsync(ct);

    return Results.Ok(request.ToDto());
}
```

### 6.4 Anti-Forgery Token Handling in the Frontend

The Next.js frontend must read the anti-forgery token from the cookie set by ASP.NET Core and include it in every state-changing request header.

```typescript
// Web/lib/api-client.ts

const XSRF_COOKIE_NAME = 'XSRF-TOKEN';
const XSRF_HEADER_NAME = 'X-XSRF-TOKEN';

function getXsrfToken(): string | undefined {
    return document.cookie
        .split('; ')
        .find(row => row.startsWith(`${XSRF_COOKIE_NAME}=`))
        ?.split('=')[1];
}

export async function apiPost<T>(path: string, body: unknown): Promise<T> {
    const response = await fetch(path, {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            [XSRF_HEADER_NAME]: getXsrfToken() ?? '',
        },
        credentials: 'include', // Required for cookie transport
        body: JSON.stringify(body),
    });

    if (!response.ok) {
        const problem = await response.json();
        throw new ApiError(response.status, problem);
    }

    return response.json() as Promise<T>;
}
```

### 6.5 Error Handling

#### 6.5.1 GlobalExceptionHandler

```csharp
// Api/Middleware/GlobalExceptionHandler.cs

using Microsoft.AspNetCore.Diagnostics;
using Microsoft.AspNetCore.Mvc;
using VacaFlow.Domain.Exceptions;

namespace VacaFlow.Api.Middleware;

public class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
        => _logger = logger;

    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext,
        Exception exception,
        CancellationToken cancellationToken)
    {
        _logger.LogError(exception, "Unhandled exception on {Method} {Path}",
            httpContext.Request.Method, httpContext.Request.Path);

        var (statusCode, title) = exception switch
        {
            DomainException       => (422, "Business rule violation"),
            ValidationException v => (400, "Validation failed"),
            AuthorizationException=> (403, "Forbidden"),
            NotFoundException     => (404, "Not found"),
            _                     => (500, "An unexpected error occurred")
        };

        var problem = new ProblemDetails
        {
            Status = statusCode,
            Title = title,
            Detail = exception is DomainException or NotFoundException or AuthorizationException
                ? exception.Message
                : null // Do not leak internal details on 500
        };

        // Attach field-level errors for validation failures
        if (exception is ValidationException ve)
            problem.Extensions["errors"] = ve.Errors;

        httpContext.Response.StatusCode = statusCode;
        await httpContext.Response.WriteAsJsonAsync(problem, cancellationToken);

        return true;
    }
}
```

## 10. Troubleshooting

### 10.1 Common Issues

| Issue | Likely Cause | Resolution |
|-------|-------------|------------|
| `dotnet run` fails immediately with "unable to open database file" | The `Api/` directory does not have write permissions for the SQLite file | Run the terminal as a user with write access to the `Api/` directory; check the `ConnectionStrings:DefaultConnection` path in `appsettings.Development.json` |
| API starts but requests return 400 on all endpoints | Anti-forgery token missing from frontend requests | Ensure the frontend reads the `XSRF-TOKEN` cookie and sends it as the `X-XSRF-TOKEN` header on every state-changing request |
| Frontend returns CORS errors in the browser console | Next.js proxy is not forwarding `/api` requests | Verify `next.config.js` has the `rewrites()` configuration pointing to the correct API port; confirm both `dotnet run` and `npm run dev` are running simultaneously |
| Seed data missing after first run | `SeedDataInitializer` threw an exception silently | Check the API console output for error messages during startup; delete `vacaflow.db` and restart the API to force a clean re-seed |
| Login succeeds but authenticated requests return 401 | Cookie not being sent — `credentials: 'include'` missing from fetch calls | Ensure all API calls in the frontend include `credentials: 'include'` in the fetch options |
| `dotnet test` fails with "No test assemblies were found" | Running from wrong directory | Run `dotnet test` from the `tests/` directory or specify the project path explicitly |
| `npm run dev` fails with "Port 3000 in use" | Another process is using port 3000 | Kill the process on port 3000 (`lsof -ti:3000 | xargs kill` on macOS/Linux) or configure Next.js to use a different port (`npm run dev -- --port 3001`) |
| EF Core migration fails with "table already exists" | An old `vacaflow.db` exists with an incompatible schema | Delete `vacaflow.db` and restart the API; migrations are idempotent against an up-to-date schema but not against a schema from a different migration baseline |
| Manager cannot see employee requests in the approval list | Employee's `ManagerId` is null — not assigned during seeding | Verify `SeedDataInitializer` sets `ManagerId` on all non-manager Employees; check the seed output in the API console; reset and re-run if needed |

## 11. Open Items and Known Constraints

### 11.1 Open Item R-01 — Manager-to-Employee Assignment Cardinality

The current implementation assumes one Manager per Employee, stored as `Employee.ManagerId` (a nullable FK to `Employee`). This is the one outstanding blocking architectural item (SAD-DD-001 §1.1, GOV-001 D002 Gate G4) and is pending formal confirmation by the Project Sponsor. The cardinality choice is isolated behind a single `Employee.ManagerId` read in `ApproveRequestSlice` and `GetRequestsSlice`, so a change to the cardinality is contained to these two components plus the EF Core model configuration and seed data.

**Impact until resolved:** Acceptance testers must confirm that the seeded Manager-Employee assignment matches the scenario used in the 11-step walkthrough.

### 11.2 MVP Scope Exclusions

The following integrations and features are explicitly deferred from the MVP and must not be added to this codebase without a formal scope change:

| Deferred Feature | Reason |
|-----------------|--------|
| Microsoft Entra ID / corporate SSO | No external identity provider in scope (NFR-COMPAT-001) |
| Email or Microsoft Teams notifications | No notification integration in MVP scope |
| Payroll, HR, or calendar system integration | No external service integrations in MVP |
| Docker containerization | Explicitly excluded to satisfy NFR-COMPAT-001 |
| Azure or cloud deployment | Local execution only for MVP |
| Mobile or native client support | Cookie-based auth requires a browser; mobile support deferred to Phase 3 (SAD-DD-001 §1.5) |

### 11.3 SQLite Production Scalability

SQLite is appropriate for single-user local execution (NFR-COMPAT-001). It does not support concurrent writes at scale. If VacaFlow_14 is deployed to a shared server with multiple simultaneous users, the persistence layer must be migrated to PostgreSQL or SQL Server (SAD-DD-001 ADR-004). EF Core's Code-First migrations and provider abstraction make this migration straightforward.
